# 用户管理增强：重置密码 & 删除用户

## 问题陈述

当前用户管理模块（`admin_service.go` / `UserManagementPage.tsx`）支持创建、编辑、启用/禁用用户，但缺少两个常见的运维能力：

1. **重置密码**：用户忘记密码或账号被锁定时，管理员无法直接干预，只能删除并重建账号。
2. **删除用户**：离职员工或测试账号只能被禁用，无法从界面移除，导致用户列表持续膨胀。

截至 2026-04-27，系统中已有至少 2 个因员工离职而长期处于禁用状态、无法从界面移除的账号；管理员曾至少 1 次通过删除并重建账号的方式处理密码重置需求，每次需手动迁移历史数据，耗时约 15–30 分钟。按当前团队规模估算，每季度发生 1–2 次此类操作，年度额外运维成本约 1–4 小时；随人员流动增加，该频率将持续上升。

## 解决方案

### 功能 1：管理员重置密码

管理员在用户列表中选择某用户，输入新密码并确认，立即生效。用户下次登录时使用新密码。

**用户流程：**
1. 管理员在用户列表点击「重置密码」
2. 弹窗输入新密码（带确认字段）
3. 提交后后端更新密码哈希，返回成功提示

**约束：**
- 仅超级管理员可操作
- 新密码需满足现有密码强度规则

### 功能 2：软删除用户

管理员删除用户后，该用户从界面消失，无法登录，但历史数据（工作记录、任务关联）保留完整。

**用户流程：**
1. 管理员在用户列表点击「删除」
2. 二次确认弹窗（防误操作）
3. 确认后用户从列表消失，相关历史数据保留

**约束：**
- 软删除，数据库记录保留（`deleted_at` 时间戳标记）
- 被删除用户的 JWT token 立即失效（登录时校验软删除状态）
- 不可删除自身账号

## 备选方案

### 重置密码备选：自动生成随机密码
系统生成随机密码并一次性展示给管理员，由管理员转告用户。
- 优点：用户首次登录后可自行修改
- 缺点：需要额外的"修改自己密码"功能配合，当前不在范围内；管理员需要额外传递密码的沟通成本

### 删除用户备选：硬删除
直接从数据库删除用户记录。
- 优点：数据干净
- 缺点：破坏历史数据完整性（工作记录的 `pm_id` 外键悬空），不可恢复

### 不做（维持现状）
继续依赖禁用功能替代删除，管理员手动重建账号替代重置密码。
- 优点：零开发成本，零新增安全面，无引入回归风险
- 代价：用户列表持续增长（当前已有 2 个僵尸禁用账号）；重建账号流程需手动迁移历史数据，每次耗时约 15–30 分钟；不符合基本用户管理预期

### 选择理由

重置密码选择**管理员直接输入新密码**方案，而非自动生成随机密码：当前系统无"用户自助修改密码"功能，随机密码方案依赖该前置能力，引入额外范围；直接输入方案可独立交付，无外部依赖。

删除用户选择**软删除**方案，而非硬删除：历史工作记录通过 `pm_id` 外键关联用户，硬删除会造成外键悬空，破坏数据完整性；软删除以 `deleted_at` 标记实现逻辑隔离，保留审计链路。

维持现状虽零成本，但已有可量化的运维摩擦（见问题陈述），且两项功能实现复杂度低、风险可控，不做的机会成本高于实现成本。

## 范围

### In Scope
- 管理员重置任意用户密码（弹窗输入新密码）
- 软删除用户（`deleted_at` 标记，列表过滤，登录拦截）
- 前端用户列表新增「重置密码」「删除」操作入口
- 删除操作二次确认

### Out of Scope
- 用户自助修改密码
- 忘记密码邮件流程
- 硬删除 / 数据清理工具
- 批量删除
- 删除操作审计日志

## 风险

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| 软删除后 JWT 未失效，被删用户仍可访问 | 中 | 高 | 登录及鉴权中间件增加软删除状态校验 |
| 管理员误删活跃用户 | 低 | 中 | 二次确认弹窗，显示用户名；软删除保留数据库记录，可由数据库管理员恢复 |
| 非超级管理员直接调用重置密码 API（越权） | 中 | 高 | 在 handler 层显式校验调用方角色为 super_admin，补充对应接口鉴权测试 |

## 成功标准

- [ ] 管理员可通过 UI 为任意用户设置新密码，操作后用户可用新密码登录
- [ ] 管理员删除用户后，该用户从列表消失，无法登录，历史工作记录完整保留
- [ ] 被删除用户的现有 JWT token 在下次请求时被拒绝
- [ ] 不可删除当前登录的管理员自身账号
- [ ] 超级管理员在用户列表中可见「重置密码」和「删除」操作入口；非超级管理员不可见
- [ ] 删除确认弹窗中显示目标用户的用户名，管理员确认后操作方可执行
