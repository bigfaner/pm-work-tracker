# Proposal: Main Item Decision Log

## Problem

项目主事项（MainItem）在推进过程中会产生大量决策：技术方案选型、资源调整、需求变更、进度计划调整等。这些决策目前没有系统化的记录方式，散落在聊天记录、邮件、会议纪要中，导致：

1. **追溯困难**：新成员加入后无法了解历史决策背景。过去半年团队新增 3 名成员，每人 onboarding 期间平均花费 2-3 天向老成员反复询问"为什么选这个方案"、"为什么延期"，占用了老成员每人约 4 小时的解释时间。
2. **重复讨论**：同样的决策点被反复讨论，浪费团队时间。近一季度回顾中发现至少 4 次跨会议重复讨论同一技术选型（如 MainItem 的排序策略、周视图的数据聚合方式），每次约 30 分钟，累计浪费约 2 小时团队会议时间。
3. **上下文缺失**：事项状态/属性变更时缺少决策依据，难以理解"为什么这样做"。近期 sprint 回顾中团队反馈：状态变更记录（StatusHistory）仅有 Remark 文本，无法追溯变更背后的决策分类和关联标签。

**紧迫性**：团队计划 Q3 扩招 2-3 人，届时 onboarding 成本将进一步放大。决策日志的缺失问题随团队规模线性增长，越早解决积累的决策记录越多，新成员上手越快。

## Solution

为主事项增加**追加式决策日志**（Decision Log），以 Timeline 形式展示在主事项详情页。

### 核心功能

- 每条决策记录包含：**预定义分类** + **自由标签** + **文本内容**
- 系统自动记录：录入人、录入时间
- **草稿→发布**生命周期：保存为草稿后可反复编辑，发布后不可修改，保留审计完整性
- 需要事项编辑权限才能添加决策

### 数据模型

```go
// DecisionLog is an append-only decision record attached to a MainItem.
// Does NOT embed BaseModel — no soft-delete needed.
// Has its own BizKey for frontend to reference specific records (e.g. edit draft).
type DecisionLog struct {
    ID          uint      `gorm:"primarykey;autoIncrement" json:"-"`
    BizKey      int64     `gorm:"not null" json:"bizKey"`
    ItemKey     int64     `gorm:"not null;index" json:"itemKey"`
    Category    string    `gorm:"type:varchar(20);not null" json:"category"`
    Tags        string    `gorm:"type:text;not null;default:''" json:"tags"`       // JSON array stored as TEXT, e.g. `["tag1","tag2"]`
    Content     string    `gorm:"type:text;not null" json:"content"`
    Status      string    `gorm:"type:varchar(10);not null;default:'draft'" json:"status"` // "draft" | "published"
    CreatedBy   int64     `gorm:"not null" json:"createdBy"`
    CreateTime  time.Time `gorm:"not null;default:CURRENT_TIMESTAMP" json:"createTime"`
    UpdateTime  time.Time `gorm:"not null;default:CURRENT_TIMESTAMP;autoUpdateTime" json:"updateTime"`
}

func (DecisionLog) TableName() string {
    return "pmw_decision_logs"
}
```

**字段说明**：
- `BizKey`：对外暴露的唯一标识符（同 ProgressRecord 模式），前端通过 BizKey 定位具体决策记录（如编辑草稿：`PUT /api/v1/teams/:teamId/main-items/:itemId/decisions/:bizKey`）
- `ItemKey`：关联 MainItem 的 BizKey（非内部 ID），遵循项目外键用 BizKey 的惯例
- `Category`：枚举值，取值范围为预定义的 6 个分类，由后端校验
- `Tags`：自由标签以 JSON 数组存储在 TEXT 列中（如 `["性能优化","缓存策略"]`），前端解析渲染；不做独立关联表，因为标签为附加属性而非独立查询维度
- `Status`：`"draft"` 或 `"published"`，字符串枚举；草稿仅 `CreatedBy` 用户可查看和编辑
- `CreatedBy`：录入人的 UserKey

**孤立草稿处理**：当草稿作者被移出团队或账号停用时，其草稿对团队成员不可见（列表查询按 `CreatedBy` 过滤），但不会自动删除。团队管理员可通过后续迭代的「管理孤立草稿」功能查看并处理这些草稿。本期不做该管理界面。

### 预定义分类

| 分类 | 说明 |
|------|------|
| technical | 技术方案选型 |
| resource | 资源/人力调整 |
| requirement | 需求变更 |
| schedule | 进度计划调整 |
| risk | 风险识别与应对 |
| other | 其他 |

用户还可在预定义分类之外输入自由标签，用于更细粒度的标记。

### 用户体验

1. 主事项详情页新增「决策记录」区域，位于子事项表格上方
2. 按时间线倒序展示已发布的决策，每条显示：分类标签、自由标签、内容摘要（截取内容前 80 个字符，超出部分以"..."省略，点击条目展开显示完整内容）、录入人、时间
3. 草稿条目以「草稿」标记显示在时间线中，仅录入人可见
4. 点击「添加决策」按钮弹出表单：分类下拉 + 标签输入 + 内容文本框
5. 表单支持「保存草稿」和「发布」两个操作
6. 草稿可点击重新编辑（分类、标签、内容均可修改），发布后不可修改
7. 权限控制：需要 `main_item:update` 权限才能添加/编辑决策

## Alternatives

### A. 状态变更时自动关联决策（扩展 StatusHistory）

在 StatusHistory 的 Remark 字段扩展为结构化决策记录。

- **Pros**：复用现有模型，无需新增数据表；决策与状态变更天然关联，减少用户操作步骤
- **Cons**：将决策与状态变更强耦合——大量决策（如技术选型、风险识别）独立于状态变更，无法归类；StatusHistory 的数据结构扩展会影响现有排序、筛选逻辑

### B. 复用现有评论/备注系统

在主事项上增加通用评论功能，用结构化文本约定（如 `@decision:technical` 前缀）标记决策。

- **Pros**：实现成本最低，仅需一个文本输入框；用户已习惯评论交互流程，无需学习新的 UI 范式，零认知负担即可上手；评论天然支持时间线展示，与现有 Remark 体验一致
- **Cons**：无法强制结构化——用户可能不遵循约定格式（漏写前缀、拼写错误），导致查询失败；分类和标签无法被程序化查询/筛选，无法统计"技术选型"类决策数量；与普通评论混在一起，决策信息容易被淹没，违背"决策可追溯"的核心诉求

### C. 不做（维持现状）

继续依赖聊天记录和会议纪要。

- **Pros**：零开发成本，团队无需适应新工具
- **Cons**：决策散落在多个渠道（聊天、邮件、会议纪要），无统一入口，搜索依赖人工回忆；随团队扩招问题加剧，onboarding 成本持续上升（如前述数据所示）；无法量化——管理层无法评估团队决策频率和类型分布，缺少改进依据

### 方案对比

| 维度 | 本方案（独立 DecisionLog） | A. 扩展 StatusHistory | B. 复用评论系统 | C. 不做 |
|------|--------------------------|----------------------|----------------|--------|
| 灵活性 | 高——独立于状态变更 | 低——仅覆盖状态变更场景 | 中——依赖约定 | — |
| 实现成本 | 中——新增模型 + API | 低——扩展现有字段 | 低——新增评论框 | 零 |
| 可查询性 | 高——结构化分类 + 标签 | 中——仅限状态变更关联 | 低——非结构化文本 | — |
| 用户采用 | 高——明确入口，低认知成本 | 中——决策时机受限 | 低——依赖用户自觉 | — |

**选择理由**：本方案在灵活性和可查询性上优于 A 和 B，实现成本可接受（约 2-3 天开发量）。决策记录的核心价值在于结构化查询和独立于状态变更的灵活性，A 和 B 都无法同时满足这两点。接受的代价是占用当前 sprint 约 2 天的容量，暂时推迟「决策通知」等增强功能。

## Scope

### In-scope

- 新增 `DecisionLog` 数据模型（含 status 字段区分草稿/已发布）
- 后端 API：创建决策 + 保存草稿 + 发布草稿 + 查询列表
- 预定义分类 + 自由标签
- 主事项详情页 Timeline 展示（草稿 + 已发布）
- 添加决策表单与交互（保存草稿 / 发布）

### Out-of-scope

- 编辑/删除已发布决策记录
- 子事项的决策日志
- 跨事项的决策搜索/筛选
- 新决策通知机制
- 决策导出功能
- 孤立草稿管理界面

### 交付计划

目标：当前 sprint 内完成交付。工作量拆分：

| 交付物 | 预估工时 |
|--------|----------|
| 数据模型 + 迁移 + Repository | 0.5 天 |
| Service + Handler API（创建/草稿/发布/列表） | 0.5 天 |
| 前端 Timeline 组件 + 列表渲染 | 0.5-1 天 |
| 前端表单组件（分类下拉 + 标签输入 + 内容）+ 集成 | 0.5-1 天 |

总计约 2-3 天开发量，由 1 名开发者连续完成。

## Risks

**评级标准**：Likelihood（6 个月内发生概率）— 高：>70%，中：30-70%，低：<30%。Impact（受影响用户比例或功能受损程度）— 高：阻塞 >50% 用户的核心使用场景，中：影响局部体验但可绕过，低：少数用户受影响且影响轻微。

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| 自由标签拼写不一致导致检索困难 | 中 | 中 | 本期在标签输入框旁显示已有标签提示（recent tags dropdown），降低拼写偏差；后续迭代加自动补全 |
| 草稿→发布后不可修改，笔误无法修正 | 低 | 低 | 草稿阶段支持反复编辑；发布前表单展示完整预览，确认后才写入；已发布记录不可修改，保持审计完整性 |
| 预定义分类不够覆盖实际场景 | 低 | 中 | 已有"其他"分类兜底；发布后第一个 sprint 内收集使用数据，若"其他"占比 >30% 则新增分类 |
| 决策日志无限增长，大量条目下查询/渲染性能下降 | 高 | 中 | 列表 API 默认分页（每页 20 条）；Timeline 组件采用懒加载（滚动加载），单次仅渲染可视区域 |
| 用户不使用决策日志，功能闲置 | 中 | 高 | 发布后首个 sprint 在主事项详情页顶部显示「决策记录 (0)」计数徽标，提醒用户该功能存在；Sprint 回顾中纳入使用率检查，活跃事项定义为「当前 sprint 内发生过状态变更或编辑的主事项」；若前两个 sprint 中活跃事项的决策记录覆盖率 < 50%，则在主事项状态变更为终态时自动弹出「是否记录决策？」引导提示 |

## Success Criteria

1. 主事项详情页可查看按时间线倒序排列的决策记录，每条显示分类标签、自由标签、内容摘要（截取前 80 字符 + "..."）、录入人、时间；点击条目可展开显示完整内容
2. 可通过表单添加决策（选择分类 + 输入标签 + 填写内容），支持「保存草稿」和「发布」两个操作
3. 草稿状态可反复编辑（分类、标签、内容均可修改），发布后不可修改
4. 草稿以「草稿」标记显示在时间线中，仅录入人可见（验证方式：用户 A 创建草稿后，用户 B 查询同一主事项的决策列表，返回结果中不含用户 A 的草稿）
5. 仅编辑权限用户可添加决策
6. 决策记录持久化存储，刷新页面不丢失
7. 列表 API 返回分页结果，每页 20 条，按时间倒序
8. 创建 API 成功时返回 201 及完整决策对象（含生成的 ID、录入人、时间戳）；必填字段（分类、内容）缺失时返回 400 及错误信息
9. 标签在决策记录列表中正确显示（分类标签 + 自由标签均可识别）
10. 分类下拉包含且仅包含 6 个预定义选项（technical, resource, requirement, schedule, risk, other），不可自定义增删
11. 自由标签以原始输入持久化：创建时输入的标签文本在刷新页面后、API 重新查询后均原样显示，不丢失、不截断
12. 已发布决策的分类、标签、内容字段均不可通过任何 API 修改；对已发布记录调用编辑接口返回 403
13. 草稿作者被移出团队后，其决策草稿不出现于任何团队成员的决策列表查询结果中，且草稿记录仍保留在数据库中（不自动删除）
