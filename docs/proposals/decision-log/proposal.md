# Proposal: Main Item Decision Log

## Problem

项目主事项（MainItem）在推进过程中会产生大量决策：技术方案选型、资源调整、需求变更、进度计划调整等。这些决策目前没有系统化的记录方式，散落在聊天记录、邮件、会议纪要中，导致：

1. **追溯困难**：新成员加入后无法了解历史决策背景
2. **重复讨论**：同样的决策点被反复讨论，浪费团队时间
3. **上下文缺失**：事项状态/属性变更时缺少决策依据，难以理解"为什么这样做"

## Solution

为主事项增加**追加式决策日志**（Decision Log），以 Timeline 形式展示在主事项详情页。

### 核心功能

- 每条决策记录包含：**预定义分类** + **自由标签** + **文本内容**
- 系统自动记录：录入人、录入时间
- 追加式（append-only），不可编辑/删除，保留完整决策历史
- 需要事项编辑权限才能添加决策

### 预定义分类

| 分类 | 说明 |
|------|------|
| technical | 技术方案选型 |
| resource | 资源/人力调整 |
| requirement | 需求变更 |
| schedule | 进度计划调整 |
| risk | 风险识别与应对 |
| other | 其他 |

用户还可在预定义分类之外输入自由标签，用于更细粒度的标记。

### 用户体验

1. 主事项详情页新增「决策记录」区域，位于子事项表格上方
2. 按时间线倒序展示，每条显示：分类标签、自由标签、内容摘要、录入人、时间
3. 点击「添加决策」按钮弹出表单：分类下拉 + 标签输入 + 内容文本框
4. 权限控制：需要 `main_item:update` 权限才能添加

## Alternatives

### A. 状态变更时自动关联决策

在 StatusHistory 的 Remark 字段扩展为结构化决策记录。但这样将决策与状态变更耦合，无法记录独立于状态变更的决策（如技术选型），不够灵活。

### B. 不做（维持现状）

继续依赖聊天记录和会议纪要。问题会随着团队规模增长而加剧，新成员 onboarding 成本高。

## Scope

### In-scope

- 新增 `DecisionLog` 数据模型
- 后端 API：创建决策 + 查询列表
- 预定义分类 + 自由标签
- 主事项详情页 Timeline 展示
- 添加决策表单与交互

### Out-of-scope

- 编辑/删除决策记录
- 子事项的决策日志
- 跨事项的决策搜索/筛选
- 新决策通知机制
- 决策导出功能

## Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| 自由标签拼写不一致导致检索困难 | 中 | 后续可加标签自动补全，本期不做 |
| 追加式不可编辑，笔误无法修正 | 低 | 可在录入前做二次确认，或后续版本支持有限编辑 |
| 预定义分类不够覆盖实际场景 | 低 | 已有"其他"分类兜底，可根据使用反馈迭代 |

## Success Criteria

1. 主事项详情页可查看按时间线排列的决策记录
2. 可通过表单添加决策（选择分类 + 输入标签 + 填写内容）
3. 仅编辑权限用户可添加决策
4. 决策记录持久化存储，刷新页面不丢失
