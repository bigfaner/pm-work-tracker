# 2.2: I-B: Custom role partial permissions + immediate effect test

## Description

在 `backend/tests/integration/rbac_permission_test.go` 中实现 `TestCustomRole_PartialPermissions`，验证自定义角色（部分权限组合）的访问控制，以及权限变更后即时生效（无缓存）。

## Reference Files

- `design/tech-design.md` § I-B — 完整流程（5 步）
- `backend/tests/integration/rbac_test.go` — `TestRoleEdit_ImmediateEffectOnNextRequest` 参考模式
- `backend/tests/integration/helpers.go` — `setupRBACTestDB`、`makeRequest`、`loginAs`

## Acceptance Criteria

- [ ] `TestCustomRole_PartialPermissions` 实现，覆盖以下 5 步流程：
  1. 创建自定义角色，仅赋予 `main_item:read` + `progress:read`
  2. 创建用户并分配该角色到 teamA
  3. 验证：`GET /main-items` → 200，`POST /main-items` → 403，`POST /archive` → 403
  4. 修改角色权限，新增 `main_item:create`
  5. 验证：`POST /main-items` → 200（同一 token，不重新登录）
- [ ] Phase 2 验证使用与 Phase 1 相同的 JWT token（不调用 `loginAs` 第二次）
- [ ] `go test ./tests/integration/ -run TestCustomRole_PartialPermissions` 通过

## User Stories

**Story 3 — 自定义角色权限组合验证**

- Given 自定义角色仅有 `main_item:read` + `progress:read`，用户绑定该角色
- When 用户调用 `GET /main-items`
- Then 返回 200

- Given 同上
- When 用户调用 `POST /main-items`（需要 `main_item:create`）
- Then 返回 403

- Given 管理员为该角色新增 `main_item:create` 权限
- When 用户使用同一 token 立即调用 `POST /main-items`（不重新登录）
- Then 返回 200（权限从 DB 实时读取，无缓存层介入）

## Implementation Notes

- 步骤 4 修改权限：通过 admin API `PUT /api/v1/admin/roles/:bizKey` 更新，或直接 DB insert `model.RolePermission`
- 推荐直接 DB insert（更简单，不依赖 admin API 的权限检查）
- 新用户需要 bcrypt hash 密码（cost=4），参考 `setupRBACTestDB` 中的模式
- `POST /main-items` body 需要完整的 `MainItemCreateReq`（title, priority, assigneeKey, startDate, expectedEndDate）
