# 2.3: I-C: Permission boundary tests (empty role, superadmin bypass, 401 vs 403)

## Description

在 `backend/tests/integration/rbac_permission_test.go` 中实现 3 个边界场景测试函数，验证权限系统的安全边界。

## Reference Files

- `design/tech-design.md` § I-C — 3 个边界场景
- `backend/tests/integration/helpers.go` — `setupRBACTestDB`、`signTokenWithClaims`、`makeRequest`
- `backend/internal/pkg/jwt/` — JWT Claims 结构
- `backend/tests/integration/auth_isolation_test.go` — 无效 token 测试参考

## Acceptance Criteria

- [ ] `TestPermBoundary_EmptyRole` — 用户绑定空权限角色（无任何 `RolePermission` 行），调用任意受保护端点 → 403
- [ ] `TestPermBoundary_SuperAdminBypass` — superadmin 调用受保护端点 → 200（不是 403/404/500）
- [ ] `TestPermBoundary_InvalidToken401` — 请求携带无效 JWT → 401（不是 403）
- [ ] `go test ./tests/integration/ -run TestPermBoundary` 全部通过（3 个测试）

## User Stories

**Story 4 — 权限边界场景验证**

- Given 用户绑定空权限角色（无任何权限码）
- When 调用任意受保护端点
- Then 返回 403

- Given superadmin 用户（绕过所有权限检查），目标资源已存在
- When 调用任意受保护端点
- Then 返回 200（不是权限拒绝；若返回 404/500 视为 fixture 缺失，不算通过）

- Given 请求携带无效 token
- When 调用任意端点
- Then 返回 401（而非 403，明确区分认证失败与授权失败）

## Implementation Notes

- `TestPermBoundary_EmptyRole`：创建角色时不插入任何 `RolePermission` 行；使用 `GET /main-items` 作为测试端点（member 通常有此权限，空角色没有）
- `TestPermBoundary_SuperAdminBypass`：使用 `setupRBACTestDB` 中已有的 superadmin 用户；测试端点使用 `GET /main-items`（需要 fixture 数据存在，否则 500）
- `TestPermBoundary_InvalidToken401`：直接构造 `Authorization: Bearer invalid-token-string`，不使用 `loginAs`；期望 401，断言 response code 不是 403
