# 1.1: Middleware: GetTeamBizKey + TeamScopeMiddleware inject int64

## Description

Rename `GetTeamID() uint` to `GetTeamBizKey() int64` in `middleware/team_scope.go`. Update `TeamScopeMiddleware` to inject `teamBizKey int64` into context instead of resolving `team.ID uint`. Remove the `teamIDUint := team.ID` line and the `c.Set("teamID", teamIDUint)` calls.

## Reference Files

- `backend/internal/middleware/team_scope.go` — source file to modify
- `backend/internal/middleware/team_scope_test.go` — test file to update
- `design/tech-design.md` §1 (Middleware context accessor) — exact before/after signatures

## Acceptance Criteria

- [ ] `GetTeamID` function removed; `GetTeamBizKey() int64` added with comma-ok assertion
- [ ] `TeamScopeMiddleware` injects `c.Set("teamBizKey", teamBizKey)` (int64) in both SuperAdmin and member paths
- [ ] `team_scope_test.go` `capturedTeamContext.teamID uint` → `teamBizKey int64`; assertions updated
- [ ] `go build ./internal/middleware/...` passes

## User Stories

**Story 3: 后端开发者新增功能时类型系统提供约束**
- Given Service 接口方法签名中，来自外部输入的 ID 参数类型为 `int64`
- When 开发者尝试将 `uint` 类型的内部 ID 直接传入该参数
- Then 编译器报类型不匹配错误，构建失败

## Implementation Notes

- The middleware still calls `teamRepo.FindByBizKey` to validate team existence — that call is unchanged
- Remove the `teamIDUint := team.ID` line entirely; inject `teamBizKey` directly
- `GetTeamBizKey` implementation per design: comma-ok assertion, return 0 on missing/wrong-type
