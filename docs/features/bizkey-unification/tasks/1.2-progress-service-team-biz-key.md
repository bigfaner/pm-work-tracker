# 1.2: ProgressService: teamID uint → teamBizKey int64 + fix TeamKey bug

## Description

Update `ProgressService` interface and implementation in `progress_service.go`:
- `Append`: `teamID uint` → `teamBizKey int64`; fix `TeamKey: int64(teamID)` → `TeamKey: teamBizKey`
- `CorrectCompletion`: `teamID uint` → `teamBizKey int64`
- `List`: `teamID uint` → `teamBizKey int64`

This is the primary data-correctness bug fix: every progress record currently stores the internal auto-increment ID in `team_key` instead of the snowflake bizKey.

## Reference Files

- `backend/internal/service/progress_service.go` — source file to modify
- `backend/internal/service/progress_service_test.go` — test file to update
- `design/tech-design.md` §2 (ProgressService) — exact before/after signatures

## User Stories

**Story 1: 进度记录写入正确的 team_key**
- Given 用户属于 bizKey 为 `123456789012345678` 的团队
- When 用户提交一条进度记录
- Then 数据库中该记录的 `team_key` 字段值为 `123456789012345678`（雪花 ID），而非内部自增 ID

## Implementation Notes

- `CorrectCompletion` and `List` receive `teamBizKey` but currently only use it for `ListBySubItem(ctx, teamID, ...)` — update that call to pass `teamBizKey` as well (repo signature accepts `uint` for teamID; this is a known mismatch in the repo layer that is out of scope — pass `uint(teamBizKey)` only if the repo interface requires it, otherwise update the repo call)
- Note: `ListBySubItem(ctx context.Context, teamID uint, subItemBizKey int64)` in the repo interface takes `uint` for teamID — this is a repo-internal concern; for now cast `uint(teamBizKey)` at the repo call site and document as a known limitation
