## Attack Points

### Attack 1: Flow Diagrams вҖ” Service-level FindByBizKey has no error branch, isPMRole flow absent

**Where**: Node J in the flowchart: `I --> J[Service еҶ…йғЁи°ғз”Ё FindByBizKey]` вҶ’ `J --> K[Repository еҶ…йғЁдҪҝз”Ё uint ID еҒҡ FK е…іиҒ”]` вҶ’ `K --> L([иҝ”еӣһз»“жһң])`. No branch exits J on failure. Also, Story 2 describes a role permission check (`isPMRole`) that rejects PM role assignment вҖ” this path has no diagram at all.

**Why it's weak**: The middleware-level `FindByBizKey` (node D) correctly shows a 404 branch, but the service-level `FindByBizKey` (node J) is a dead end вҖ” no error path. If the team record disappears between middleware and service execution (race condition, soft-delete), the diagram gives no guidance. The `isPMRole` path is the only user-visible business rule in the entire PRD (Story 2 AC: "зі»з»ҹжӯЈзЎ®иҜҶеҲ«иҜҘи§’иүІдёә PM и§’иүІе№¶иҝ”еӣһ `ErrCannotAssignPMRole` й”ҷиҜҜ") yet it has no flow representation whatsoever.

**What must improve**: Add an error branch from node J (e.g., `J -->|и®°еҪ•дёҚеӯҳеңЁ| M[иҝ”еӣһ 500 / ErrTeamNotFound]`). Add a second flowchart covering the `InviteMember` вҶ’ `isPMRole` decision path, showing the `ErrCannotAssignPMRole` branch.

### Attack 2: User Stories вҖ” "еӣўйҳҹ PM" user type undefined in background section

**Where**: prd-user-stories.md Story 2: `**As a** еӣўйҳҹ PM`. "еӣўйҳҹ PM" appears nowhere in the background section.

**Why it's weak**: The rubric requires "one story per target user" where target users are defined in the background section. "еӣўйҳҹ PM" is a third user type introduced only in the user stories file with no corresponding definition in В§йңҖжұӮиғҢжҷҜ дәәе‘ҳ. A reader of the spec alone would not know that team PMs are affected. This is a direct inconsistency between sections (-3 pts per the deduction rules), softened here because the story itself is coherent вҖ” the inconsistency is a documentation gap, not a logical contradiction.

**What must improve**: Add "еӣўйҳҹ PM" to В§йңҖжұӮиғҢжҷҜ дәәе‘ҳ with a one-line description of how they are affected (e.g., "еӣўйҳҹ PMпјҡи§’иүІжқғйҷҗеҲӨж–ӯдҫқиө– bizKey жӯЈзЎ®жҖ§пјҢдҝ®еӨҚеҗҺ isPMRole еҲӨж–ӯдёҚеҶҚеӣ  ID зұ»еһӢж··з”ЁиҖҢиҜҜеҲӨ").

### Attack 3: Scope Clarity вҖ” 7 handler files in scope but functionally undocumented

**Where**: В§Scope In Scope: "Handler и°ғз”ЁзӮ№пјҲ7 дёӘж–Үд»¶пјүпјҡ`item_pool_handler.go`гҖҒ`main_item_handler.go`гҖҒ`progress_handler.go`гҖҒ`report_handler.go`гҖҒ`sub_item_handler.go`гҖҒ`team_handler.go`гҖҒ`view_handler.go` вҖ” д»Һ context/иҜ·жұӮдёӯдј йҖ’ bizKeyпјҢиҖҢйқһи§ЈжһҗеҗҺзҡ„ uint ID". В§еҠҹиғҪжҸҸиҝ° table: 14 rows, zero rows for any handler file. Row 1 says "жүҖжңү handler и°ғз”Ёж–№йңҖеҗҢжӯҘжӣҙж–°" but names no specific handler or method.

**Why it's weak**: The scope makes a concrete promise (7 named files will change) that the functional spec does not fulfill. A developer reading only В§еҠҹиғҪжҸҸиҝ° cannot determine which methods in `report_handler.go` change or what the before/after looks like. "жүҖжңү handler и°ғз”Ёж–№йңҖеҗҢжӯҘжӣҙж–°" is a blanket statement, not a functional spec. If a handler has both a `teamID uint` path and a `teamBizKey int64` path, the spec gives no guidance on which to change.

**What must improve**: Add one row per handler file to the functional spec table, or add a note explicitly stating "handler changes are mechanical call-site updates вҖ” see service rows 2вҖ“14 for the corresponding signature changes, no handler-specific logic changes". Either approach resolves the scope-spec gap.
