# 4.2: Role & Permission Handlers

## Description

Implement HTTP handlers for role CRUD endpoints and permission query endpoints. Register routes in the router with `RequirePermission("user:manage_role")` for admin endpoints.

## Reference Files

- API Handbook: `docs/features/rbac-permissions/design/api-handbook.md` — all new endpoints
- Tech Design: `docs/features/rbac-permissions/design/tech-design.md` — "Router Changes" table
- Target: `internal/handler/role_handler.go` (new), `internal/handler/permission_handler.go` (new), `internal/handler/router.go` (modify)

## Acceptance Criteria

- [ ] `GET /api/v1/admin/roles` — list roles with search, is_preset filter, pagination
- [ ] `POST /api/v1/admin/roles` — create role with name, description, permission_codes
- [ ] `GET /api/v1/admin/roles/:id` — get role detail with permissions list
- [ ] `PUT /api/v1/admin/roles/:id` — update role (preset rules enforced by service)
- [ ] `DELETE /api/v1/admin/roles/:id` — delete role (usage check in service)
- [ ] `GET /api/v1/admin/permissions` — list all permission codes grouped by resource
- [ ] `GET /api/v1/me/permissions` — current user's permissions (auth only, no specific permission code required)
- [ ] All admin routes protected by `RequirePermission("user:manage_role")`
- [ ] `/me/permissions` requires only `AuthMiddleware` (no permission code)
- [ ] Request validation returns `ERR_VALIDATION` (422) for invalid input
- [ ] Integration tests using `net/http/httptest` cover each endpoint

## Implementation Notes

- Follow existing handler patterns in the codebase (check `internal/handler/` for conventions)
- Use the same response envelope: `{"code": 0, "data": {...}}` for success
- Pagination follows existing pattern (check other list endpoints)
- `/me/permissions` returns `UserPermissionsResp` with `is_superadmin` and `team_permissions` map
- `/admin/permissions` returns the static registry data — no DB query needed
