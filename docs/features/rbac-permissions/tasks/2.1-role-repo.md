# 2.1: RoleRepo Implementation

## Description

Implement the `RoleRepo` interface for role CRUD, permission binding management, usage counting, and cross-team permission checking. This is the data access layer for all role-related operations.

## Reference Files

- Tech Design: `docs/features/rbac-permissions/design/tech-design.md` — "RoleRepo Interface" section
- Target: `internal/repository/role_repo.go` (interface), `internal/repository/gorm/role_repo.go` (implementation), tests

## Acceptance Criteria

- [ ] `List(ctx)` — returns all roles ordered by created_at
- [ ] `FindByID(ctx, id)` — returns role or error
- [ ] `FindByName(ctx, name)` — returns role or error (for duplicate name check)
- [ ] `Create(ctx, role)` — creates role with GORM
- [ ] `Update(ctx, role)` — updates role fields
- [ ] `Delete(ctx, id)` — soft delete (sets deleted_at)
- [ ] `ListPermissions(ctx, roleID)` — returns `[]string` of permission codes for a role
- [ ] `SetPermissions(ctx, roleID, codes)` — replaces all permission bindings for a role (delete old + insert new in transaction)
- [ ] `CountMembersByRoleID(ctx, roleID)` — counts `team_members` using this role_id
- [ ] `HasPermission(ctx, userID, code)` — checks if ANY of user's team member roles has the given permission code (non-team-context check)
- [ ] Unit tests with `:memory:` SQLite cover all methods including edge cases

## Implementation Notes

- Follow existing repo patterns in the codebase (check `internal/repository/gorm/` for conventions)
- `SetPermissions` must be atomic — delete all existing then insert new within a transaction
- `HasPermission` joins `team_members` → `role_permissions` to check if user has the code in any team
- Use `context.Context` throughout for consistency
