# 4.1: Permission Middleware

## Description

Implement the `RequirePermission(code)` middleware and update `AuthMiddleware` and `TeamScopeMiddleware` to support the new RBAC permission checking flow. This is the core enforcement layer.

## Reference Files

- Tech Design: `docs/features/rbac-permissions/design/tech-design.md` — "Middleware Chain" section
- Target: `internal/middleware/permission.go` (new), `internal/middleware/auth.go` (modify), `internal/middleware/team_scope.go` (modify), tests

## Acceptance Criteria

- [ ] `RequirePermission(code)` middleware: checks if user has the specified permission code
  - SuperAdmin → always pass (c.Next())
  - Team context (permCodes set) → code in permCodes → pass / 403
  - Non-team context → query RoleRepo.HasPermission(userID, code) → pass / 403
- [ ] `AuthMiddleware` updated: after JWT extraction, load User from DB and set `isSuperAdmin` in gin context
- [ ] `TeamScopeMiddleware` updated: load TeamMember.RoleID → load RolePermissions → set `permCodes []string` in context
  - If SuperAdmin → set empty permCodes (bypass handled in RequirePermission)
  - If not team member → 403
- [ ] Context helper functions: `GetUserID()`, `IsSuperAdmin()`, `GetPermCodes()`, `GetTeamID()`
- [ ] Error response format: `{"code": "ERR_FORBIDDEN", "message": "权限不足：缺少 {code} 权限"}`
- [ ] Unit tests with mock gin context cover: superadmin bypass, team-scoped pass/deny, non-team pass/deny, non-member 403

## Implementation Notes

- `RequirePermission` returns a `gin.HandlerFunc` closure that captures the permission code
- The middleware chain is: AuthMiddleware → [TeamScopeMiddleware] → RequirePermission → Handler
- TeamScopeMiddleware only runs on routes with `:teamId` param; other routes skip it
- Permission codes are loaded from DB on every request (no cache for now — PRD says < 10ms, SQLite local query is fast enough)
- Remove old `RequireRole()` and `RequireTeamRole()` from `middleware/rbac.go` (or mark deprecated if other code still references them — clean removal in task 4.3)
