# 1.2: Data Models (Role, RolePermission, TeamMember Changes)

## Description

Add new `Role` and `RolePermission` GORM models. Modify `TeamMember` model to add `RoleID` field (keeping existing `Role` string field for migration transition). Update `User` model to keep `IsSuperAdmin` and prepare for `CanCreateTeam` removal after migration.

## Reference Files

- Tech Design: `docs/features/rbac-permissions/design/tech-design.md` — "Data Models" section and ER Diagram
- Existing models: `internal/model/` directory
- Target: `internal/model/role.go` (new), `internal/model/team_member.go` (modify), `internal/model/user.go` (modify)

## Acceptance Criteria

- [ ] `Role` model with fields: ID, Name (unique), Description, IsPreset, CreatedAt, UpdatedAt, DeletedAt
- [ ] `RolePermission` model with fields: ID, RoleID (FK), PermissionCode, unique constraint on (RoleID, PermissionCode)
- [ ] `TeamMember` model gains `RoleID *uint` field (nullable for migration transition)
- [ ] `User` model keeps `IsSuperAdmin` bool; `CanCreateTeam` remains temporarily (removed in migration task)
- [ ] GORM table names: `roles`, `role_permissions`
- [ ] No behavioral changes to existing code — only model struct additions

## Implementation Notes

- Use GORM soft delete (`DeletedAt`) for Role model only
- RolePermission does NOT need soft delete (hard delete when updating role permissions)
- TeamMember.RoleID is nullable initially; migration (1.3) will populate it then the old `Role` string field becomes unused
- Follow existing model conventions in the codebase for struct tags and naming
