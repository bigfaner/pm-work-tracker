# 3.1: RoleService Implementation

## Description

Implement the `RoleService` interface with all business logic for role management: CRUD with preset role protection, permission code validation, name conflict detection, and user permissions aggregation.

## Reference Files

- Tech Design: `docs/features/rbac-permissions/design/tech-design.md` — "RoleService Interface" section
- API Handbook: `docs/features/rbac-permissions/design/api-handbook.md` — error codes and response shapes
- Target: `internal/service/role_service.go`, `internal/service/role_service_test.go`

## Acceptance Criteria

- [ ] `ListRoles(ctx)` — returns all roles with permission_count and member_count
- [ ] `GetRole(ctx, roleID)` — returns role detail with full permission list
- [ ] `CreateRole(ctx, req)` — validates name (2-50 chars, unique), validates all codes via `permissions.ValidateCode()`, at least 1 code; returns created role
- [ ] `UpdateRole(ctx, roleID, req)` — preset role protection (superadmin immutable, pm/member can change description + permissions but not name); validates codes
- [ ] `DeleteRole(ctx, roleID)` — preset role protection; checks usage count > 0 → ERR_ROLE_IN_USE
- [ ] `ListPermissionCodes(ctx)` — returns `permissions.Registry` directly
- [ ] `GetUserPermissions(ctx, userID)` — returns `UserPermissions{IsSuperAdmin, TeamPermissions map}` by joining team_members → roles → role_permissions
- [ ] Correct error codes: ERR_ROLE_NOT_FOUND, ERR_ROLE_NAME_EXISTS, ERR_ROLE_IN_USE, ERR_PRESET_ROLE_IMMUTABLE, ERR_INVALID_PERMISSION_CODE
- [ ] Unit tests with mocked RoleRepo cover all business rules

## Implementation Notes

- Service layer owns business rules; repo layer is a thin data access wrapper
- `GetUserPermissions` needs to query: user's IsSuperAdmin flag + all team_members entries for the user + their role's permission codes
- For `GetUserPermissions`, use a single efficient query or build the map in Go after loading data
- Mock the RoleRepo interface in tests using testify/mock or manual mocks
