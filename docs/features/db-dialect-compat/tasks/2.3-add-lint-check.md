# 2.3: Add SQLite Keyword Lint Check

## Description

Add a `check_sqlite_keywords` function to `scripts/lint-staged.sh` that detects hardcoded SQLite-only SQL syntax in the repository layer. This prevents future developers from accidentally introducing new SQLite-specific SQL.

## Reference Files

- `docs/features/db-dialect-compat/design/tech-design.md` — Lint Check Specification (Story 3)
- `scripts/lint-staged.sh` — existing lint script

## Acceptance Criteria

- [ ] `check_sqlite_keywords` function added to `lint-staged.sh`
- [ ] Detects patterns: `SUBSTR(`, `CAST(.+AS INTEGER)`, `datetime(.now.)`, `pragma_table_info`
- [ ] Scan scope: `backend/internal/repository/gorm/*.go` only (excluding `*_test.go`)
- [ ] Hardcoded SQLite SQL in repo layer → commit blocked with clear error message
- [ ] Code using `dialect.CastInt()` / `dialect.Substr()` / `dialect.Now()` → commit passes
- [ ] Function is called from the main lint flow

## User Stories

**Story 3: 开发者编写新 SQL 时被自动化检查拦截**
- Given 开发者在 `backend/internal/repository/` 下的 `.go` 文件中写入了包含 SQLite 专属 SQL
- When 执行 git commit 触发 lint-staged
- Then 提交被拦截，提示使用 `dialect` 包
- Given 开发者在 repo 层使用 `dialect.CastInt()` 等方法
- When 执行 git commit
- Then 提交正常通过

## Implementation Notes

Implementation from design:

```bash
check_sqlite_keywords() {
    local pattern='SUBSTR\(|CAST.*AS INTEGER|datetime(.now.)|pragma_table_info'
    local files=$(git diff --cached --name-only -- 'backend/internal/repository/gorm/*.go' | grep -v '_test.go')
    if [ -n "$files" ]; then
        if grep -En "$pattern" $files; then
            echo "ERROR: Hardcoded SQLite syntax found in repo layer."
            echo "Use pkg/dbutil.Dialect instead. See docs/features/db-dialect-compat/."
            return 1
        fi
    fi
    return 0
}
```

Known limitation: grep cannot distinguish comments from code — may produce false positives for comments containing keywords. For legitimate keyword usage, add `# nosqlite` marker at end of line.
