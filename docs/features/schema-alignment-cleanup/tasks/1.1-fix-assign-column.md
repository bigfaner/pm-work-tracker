# 1.1: Fix Assign column name

## Description

Fix P0 bug where `SubItem.Assign()` writes to non-existent `assignee_id` column instead of `assignee_key`. The assignment silently fails, leaving `assignee_key` unchanged.

## Reference Files

- `backend/internal/service/sub_item_service.go:262` — the buggy `UpdateFields` call
- `docs/features/schema-alignment-cleanup/design/tech-design.md` — Item 1 specification

## Acceptance Criteria

- [ ] `grep -rn "assignee_id" backend/internal/service/sub_item_service.go` returns zero results
- [ ] `go test ./internal/service/ -run TestSubItem` passes
- [ ] Test asserts `fields["assignee_key"]` exists and `fields["assignee_id"]` does not

## User Stories

**Story 1: 指派子项功能恢复**
- Given 一个未指派的子项
- When 用户选择一个成员并点击"指派"
- Then 子项的 `assignee_key` 列被正确更新（而非错误地写入 `assignee_id` 列）
- And 刷新页面后，指派人名称仍然显示在子项上

## Implementation Notes

Single-line fix in `sub_item_service.go:262`: change `"assignee_id"` to `"assignee_key"` in the `UpdateFields` call map. Write a test first that asserts the correct field name, then fix.
