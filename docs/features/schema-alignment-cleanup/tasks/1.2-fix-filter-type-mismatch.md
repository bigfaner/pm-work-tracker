# 1.2: Fix filter type mismatch

## Description

Fix P0 bug where `assignee_key` filter in `filter_helpers.go` passes a raw `*string` bizKey directly into SQL `WHERE` clause comparing against a `BIGINT` column. The type mismatch causes the filter to match zero rows in MySQL (or silently return all items), creating an authorization bypass.

## Reference Files

- `backend/internal/repository/gorm/filter_helpers.go:15` — the buggy filter logic
- `docs/features/schema-alignment-cleanup/design/tech-design.md` — Item 2 specification, error handling section

## Acceptance Criteria

- [ ] `go test ./internal/repository/gorm/ -run TestFilter` passes
- [ ] Filter with valid string bizKey returns correct subset (not all, not empty)
- [ ] Filter with invalid string returns zero results (`WHERE 1 = 0`)
- [ ] Behavior is consistent across MySQL and SQLite

## User Stories

**Story 2: 按人筛选功能恢复**
- Given 项目列表页面，存在多个已指派给不同成员的项目
- When 用户从"按人筛选"下拉框选择某个成员
- Then 列表仅显示指派给该成员的项目（而非返回全部或空结果）
- And 该行为在 MySQL 和 SQLite 上均一致

## Implementation Notes

Add `strconv.ParseInt` conversion before SQL query. On parse failure, apply `WHERE 1 = 0` (fail-closed) instead of skipping the filter. See tech-design.md Item 2 for exact code. Write failing test first for both valid and invalid input cases.
