# 2.5: SQL Pushdown — GanttView + Minor Performance Fixes

## Description

Push status filter to SQL for GanttView. Add composite index on `sub_items(team_id, main_item_id)` if benchmarking shows improvement. Fix `TeamService.GetTeamDetail` to use `COUNT(*)` instead of loading full member list. Add LRU eviction to `linkageMuMap`.

## Reference Files

- `backend/internal/service/view_service.go` — `GanttView` (line ~474)
- `backend/internal/repository/gorm/sub_item_repo.go` — potential index
- `backend/internal/service/team_service.go` — `GetTeamDetail`
- `backend/internal/service/view_service.go` — `linkageMuMap`

## Acceptance Criteria

- [ ] `GanttView` calls `mainItemRepo.ListByTeamAndStatus` when status filter is set, falls back to existing method when no filter
- [ ] `GetTeamDetail` uses `COUNT(*)` for member count instead of loading all members
- [ ] `linkageMuMap` has bounded capacity of 1000 entries with LRU eviction
- [ ] Composite index added only if `EXPLAIN` shows full table scan AND benchmark shows >20ms improvement on 200-record dataset
- [ ] `go test ./...` passes

## Implementation Notes

For GanttView, the status filter is the only filter that can be pushed to SQL cleanly. Other filters (date range) may remain in Go. For the index decision: first run `EXPLAIN` on the current sub_items query, then benchmark with and without the index. Only add if the improvement is significant. For `linkageMuMap`, track insertion order with a separate slice or use a simple timestamp-based eviction.
