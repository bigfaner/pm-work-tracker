# 3.4: Create StatusTransitionDropdown Component

## Description

Extract the ~600 lines of duplicated status dropdown logic across 5 inline implementations into a single shared `StatusTransitionDropdown` component. Supports both main item and sub-item types with an optional `onBeforeTerminalStatus` callback for sub-item's achievement dialog.

## Reference Files

- `frontend/src/components/shared/StatusTransitionDropdown.tsx` — new file
- `frontend/src/components/shared/StatusTransitionDropdown.test.tsx` — new test
- `frontend/src/pages/ItemViewPage.tsx` — reference implementations (lines 1242-1459)
- `frontend/src/pages/MainItemDetailPage.tsx` — reference implementations (lines 701-927)
- `frontend/src/pages/SubItemDetailPage.tsx` — reference implementation (lines 425-592)

## Acceptance Criteria

- [ ] `StatusTransitionDropdown` component with props: `currentStatus`, `itemType`, `teamId`, `itemId`, `onStatusChanged`, `onBeforeTerminalStatus?`, `disabled?`
- [ ] Fetches available transitions from API
- [ ] Shows confirmation dialog for terminal status transitions
- [ ] `onBeforeTerminalStatus` callback fires before terminal transition (used by sub-items for achievement dialog)
- [ ] Co-located test file covers: rendering, transition fetching, confirmation dialog, callback invocation
- [ ] `npm test` passes

## Implementation Notes

The component uses the status transition API to fetch available transitions. When a terminal status is selected: (1) if `onBeforeTerminalStatus` is provided, call it and wait for `true`/`false`; (2) if callback returns `true` or is absent, show built-in confirmation dialog; (3) on confirm, call the status change API. The `itemType` prop determines which API endpoint to call (main vs sub item status change).
