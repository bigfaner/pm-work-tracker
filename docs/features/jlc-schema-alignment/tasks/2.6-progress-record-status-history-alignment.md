# 2.6: ProgressRecord + StatusHistory 表对齐（append-only model + repo）

## Description

以 `pmw_progress_records` + `pmw_status_histories` 两张 append-only 表为单位完成对齐。这两张表不嵌入 `BaseModel`（无 biz_key、无软删字段），字段结构独立。

**Model（`backend/internal/model/progress_record.go`）**：
- 不嵌入 `BaseModel`，手动声明 `ID`、`CreateTime`
- FK 字段：`SubItemID uint` → `SubItemKey int64`，`TeamID uint` → `TeamKey int64`，`AuthorID uint` → `AuthorKey int64`
- `IsPmCorrect int`（`TINYINT(1)`，非 bool）
- `TableName()` → `pmw_progress_records`

**Model（`backend/internal/model/status_history.go`）**：
- 不嵌入 `BaseModel`，手动声明 `ID`、`CreateTime`
- `ItemID int64` → `ItemKey int64`（`json:"itemKey"`）
- `ChangedBy int64`（`json:"changedBy"`）
- `TableName()` → `pmw_status_histories`

**Repo 实现**：
- `progress_record_repo.go`：更新字段引用（`SubItemKey`、`TeamKey`、`AuthorKey`）
- `status_history_repo.go`：更新字段引用（`ItemKey`、`ChangedBy`）
- 两张表无软删，无 `NotDeleted` scope，无 `SoftDelete` 方法

**Service 层**：
- `progress_service.go`：`Create()` 赋值 `SubItemKey`、`TeamKey`、`AuthorKey`（从 bizKey 获取）
- `main_item_service.go` 中 `ChangeStatus()` 写入 `StatusHistory` 时使用 `ItemKey`

## Acceptance Criteria

- [ ] `ProgressRecord` 不嵌入 `BaseModel`，含 `SubItemKey int64`、`TeamKey int64`、`AuthorKey int64`
- [ ] `StatusHistory` 不嵌入 `BaseModel`，含 `ItemKey int64`、`ChangedBy int64`
- [ ] `ProgressRecord.TableName()` 返回 `pmw_progress_records`
- [ ] `StatusHistory.TableName()` 返回 `pmw_status_histories`
- [ ] 两张表的 repo 无 `SoftDelete`、无 `NotDeleted` scope
- [ ] `go test ./backend/internal/...` 中 progress/status_history 相关测试通过

## User Stories

Story 1（MySQL schema 执行）：model 字段与 pmw_progress_records、pmw_status_histories 表字段一一对应。

## Implementation Notes

- `IsPmCorrect` 类型为 `int`（对应 `TINYINT(1)`），不使用 `bool`
- `StatusHistory.ItemKey` 存储的是被变更记录的 bizKey（非内部 id）
- `ChangedBy` 存储操作者的 bizKey（非内部 id）
- append-only 表只有 INSERT，无 UPDATE/DELETE，无需软删逻辑
