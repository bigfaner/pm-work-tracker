# 2.4a: SubItem model + repo 对齐

## Description

以 `pmw_sub_items` 表为单位，完成 model 和 repo 层对齐。

**Model（`backend/internal/model/sub_item.go`）**：
- 嵌入新 `BaseModel`
- `Status` → `ItemStatus`（`json:"itemStatus"`）
- `Description` → `ItemDesc`（`json:"itemDesc"`）
- `StartDate` → `PlanStartDate`（`json:"planStartDate"`）
- FK 字段：`TeamID uint` → `TeamKey int64`，`MainItemID uint` → `MainItemKey int64`，`AssigneeID *uint` → `AssigneeKey *int64`
- `TableName()` → `pmw_sub_items`

**Repo 实现（`backend/internal/repository/gorm/sub_item_repo.go`）**：
- 所有 `Find`/`First`/`List`/`Count` 加 `.Scopes(NotDeleted)`
- 实现 `FindByBizKey(ctx, bizKey int64)`
- 实现 `SoftDelete(ctx, id uint)`：`UPDATE SET deleted_flag=1, deleted_time=NOW() WHERE id=? AND deleted_flag=0`，`RowsAffected==0` 返回 `ErrNotFound`
- 更新 `List`、`ListByMainItem`、`ListByTeam` 中的字段引用

## User Stories

Story 2（软删除接口）：
- Given `subItemRepo.SoftDelete(ctx, id)` 被调用
- Then 数据库执行 UPDATE SET deleted_flag=1，记录不被物理删除

Story 3（自动过滤软删记录）：
- Given 已软删的子事项
- When 调用 `subItemRepo.List`
- Then 不返回该子事项

## Implementation Notes

- `SoftDelete` 需检查 `RowsAffected`，为 0 时返回 `apperrors.ErrNotFound`（幂等保证）
- `MainItemKey` 引用 MainItem 的 bizKey（int64），依赖 2.3a 完成后的 model 定义
- 本任务完成后 service/handler 层可能有编译错误，属预期状态，由 2.4b 修复
