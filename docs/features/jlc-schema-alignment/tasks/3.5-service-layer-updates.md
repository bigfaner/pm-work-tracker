# 3.5: Service 层：biz_key 赋值 + SoftDelete 调用 + GetByBizKey

## Description

更新所有受影响的 service 文件，完成三类变更：
1. `Create()` 方法赋值 `BizKey: snowflake.Generate()`，并捕获 MySQL 1062 错误转换为 `ErrDuplicateBizKey`
2. `Delete()` 方法改为调用 repo 的 `SoftDelete()`
3. 新增 `GetByBizKey()` 方法，同步更新 service 接口定义

受影响的 service 文件：
- `main_item_service.go`：`Create` 赋值 BizKey，新增 `GetByBizKey`
- `sub_item_service.go`：`Create` 赋值 BizKey，`Delete` 改调 `SoftDelete`，新增 `GetByBizKey`
- `team_service.go`：`CreateTeam` 赋值 BizKey，`DisbandTeam` 改调 `SoftDelete`，`RemoveMember` 改调 `teamMemberRepo.SoftDelete`，新增 `GetTeamByBizKey`
- `item_pool_service.go`：`Submit` 赋值 BizKey，新增 `GetByBizKey`
- `auth_service.go`：`Register` 赋值 BizKey

- `design/tech-design.md` §Interfaces 5, 5b — Service 层变更和接口定义

## Acceptance Criteria

- [ ] 所有 `Create()` 方法在构造 model 时赋值 `BizKey: snowflake.Generate()`
- [ ] `Create()` 捕获 MySQL 1062 错误，返回 `apperrors.ErrDuplicateBizKey`
- [ ] `sub_item_service.Delete()` 调用 `subItemRepo.SoftDelete()`
- [ ] `team_service.DisbandTeam()` 调用 `teamRepo.SoftDelete()`
- [ ] `team_service.RemoveMember()` 调用 `teamMemberRepo.SoftDelete()`
- [ ] 5 个 service 接口均新增对应 `GetByBizKey` 方法（见 tech-design §5b）
- [ ] `go test ./internal/service/...` 通过

## User Stories

Story 1（schema 执行）：`biz_key` 字段在 Create 时由雪花算法赋值，确保每条记录有唯一业务键。
Story 2（软删接口）：service 层 Delete 统一调用 repo 的 SoftDelete，不直接调用 `db.Delete()`。
Story 6（bizKey 导航）：service 层新增 `GetByBizKey`，供 handler 层调用。

## Implementation Notes

- `auth_service.Register` 赋值 BizKey 时，User model 的 BizKey 字段需在 `Register` 方法中设置，不依赖 GORM 自动填充
- `isMySQLDuplicateError` 辅助函数在 3.2 任务中已实现，直接调用
- service 接口变更后，所有 mock 实现（测试文件中）需同步添加新方法
- `item_pool_service.GetByBizKey` 签名为 `GetByBizKey(ctx context.Context, teamID uint, bizKey int64)`（含 teamID 权限校验）
