# 3.6: Handler 层：路径参数 uint→int64 + GetByBizKey 调用

## Description

更新所有 handler 文件，将路径参数解析从 `uint` 改为 `int64`（bizKey），并将 `GetByID` 调用改为 `GetByBizKey`。DELETE handler 需两步：先用 bizKey 查出记录，再用内部 id 执行删除。

受影响的 handler 文件（所有有 `:itemId`/`:teamId` 路径参数的 handler）：
- `main_item_handler.go`
- `sub_item_handler.go`
- `team_handler.go`
- `item_pool_handler.go`
- `user_handler.go`（如有 bizKey 路径参数）

## Reference Files

- `design/tech-design.md` §Interfaces 3b — Handler 层变更示例代码
- `backend/internal/handler/` 目录下所有受影响 handler 文件

## Acceptance Criteria

- [ ] 所有 handler 路径参数解析使用 `strconv.ParseInt(c.Param("itemId"), 10, 64)`
- [ ] GET/PUT handler 调用 `svc.GetByBizKey(ctx, bizKey)` 而非 `svc.GetByID(ctx, uint(id))`
- [ ] DELETE handler 先调用 `FindByBizKey` 获取记录，再用 `item.ID` 调用 `svc.Delete(ctx, item.ID)`
- [ ] `bizKey` 解析失败（非数字）返回 400 错误
- [ ] `go test ./internal/handler/...` 通过

## User Stories

Story 6（bizKey 导航）：
- Given 后端 API 端点（如 `GET /api/v1/teams/:teamId/main-items/:itemId`）
- When 接收到以 `bizKey` 为值的路径参数
- Then 后端通过 `FindByBizKey` 正确定位记录并返回响应

Story 4（前端消费更新后的 API 字段）：
- Given 后端已部署新版本 API
- When 前端调用任意资源接口
- Then 响应体中存在 `bizKey` 字段，不存在 `id` 字段

## Implementation Notes

- DELETE handler 两步模式：`FindByBizKey` → `Delete(ctx, item.ID)`，service `Delete` 签名不变（仍接受 `uint` id）
- `ParseInt` 失败时用 `apperrors.RespondError(c, apperrors.ErrValidation)` 返回 400
- handler 测试中 mock service 需同步添加 `GetByBizKey` 方法
