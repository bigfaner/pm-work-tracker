# 2.4b: SubItem service + handler 对齐

## Description

以 `pmw_sub_items` 表为单位，完成 service 和 handler 层对齐。依赖 2.4a（model + repo 已对齐）。

**Service（`backend/internal/service/sub_item_service.go`）**：
- `Create()` 赋值 `BizKey: snowflake.Generate()`，捕获 MySQL 1062 返回 `ErrDuplicateBizKey`
- `Delete()` 内部改调 `subItemRepo.SoftDelete()`（接口签名不变：`Delete(ctx, teamID, callerID, itemID uint)`）
- 新增 `GetByBizKey(ctx context.Context, bizKey int64) (*model.SubItem, error)`
- 更新 `SubItemService` 接口定义（新增 `GetByBizKey`）

**Handler（`backend/internal/handler/sub_item_handler.go`）**：
- `:subItemId` 路径参数解析改为 `strconv.ParseInt`（bizKey int64）
- GET/PUT 调用 `svc.GetByBizKey(bizKey)`
- DELETE：先 `svc.GetByBizKey(bizKey)` 取 record，再 `svc.Delete(teamID, callerID, record.ID)`

## User Stories

Story 2（软删除接口）：
- Given 调用 `subItemService.Delete()`
- When 执行操作
- Then 数据库执行 UPDATE SET deleted_flag=1，记录不被物理删除

Story 6（bizKey 资源导航）：
- Given `DELETE /api/v1/teams/:teamId/sub-items/:subItemId`（subItemId 为 bizKey）
- When 接收请求
- Then 通过 `GetByBizKey` 定位记录，再执行软删

## Implementation Notes

- `SubItemService.Delete()` 签名不变（仍接受 uint），内部改调 `SoftDelete`，对 handler 透明
- mock 测试文件中 `SubItemService` 的 mock 实现需同步添加 `GetByBizKey` 方法
