# 4.5: ItemPoolService Implementation

## Description
Implement `ItemPoolService` with Submit, Assign (atomic transaction), Reject, List, and Get methods.

## Reference Files
- `design/tech-design.md` — Interfaces: ItemPoolService
- `design/tech-design.md` — Key Constraints Summary (ItemPool assign → creates SubItem atomically)
- `api-handbook.md` — ItemPool section (ITEM_ALREADY_PROCESSED, ITEM_NOT_FOUND errors)

## Acceptance Criteria
- `Submit`: creates `ItemPool` record with status `待分配`, `SubmitterID = submitterID`
- `Assign`: runs in a DB transaction — updates `ItemPool.Status = 已分配`, sets `AssignedMainID`, `AssignedSubID`, `ReviewerID`, `ReviewedAt`; creates a new `SubItem` under the specified `MainItem`; if SubItem creation fails, transaction rolls back
- `Assign`: returns `ITEM_ALREADY_PROCESSED` error if pool item status is not `待分配`
- `Reject`: updates `ItemPool.Status = 已拒绝`, sets `RejectReason`, `ReviewerID`, `ReviewedAt`; returns `ITEM_ALREADY_PROCESSED` if not `待分配`
- `List`: paginates with `ItemPoolFilter.Status` filter
- Unit test for `Assign`: verifies rollback when SubItem creation fails (mock SubItemRepo to return error)

## Implementation Notes
- `Assign` transaction must use the same `*gorm.DB` instance for both operations — pass `db` to constructor
- The SubItem created by `Assign` inherits `Title`, `Background` (as Description) from the pool item
- Constructor: `NewItemPoolService(poolRepo, subItemRepo, mainItemRepo, db) ItemPoolService`
