# 11.7: Backend Integration Tests: Progress Append → MainItem Completion Update

## Description
Write integration tests verifying the end-to-end flow from progress append through SubItem completion sync to MainItem completion rollup, plus the ItemPool assign transaction.

## Reference Files
- `design/tech-design.md` — Testing Strategy / Integration Tests (progress append → MainItem completion, ItemPool assign transaction)
- `design/tech-design.md` — Key Constraints Summary

## Acceptance Criteria
- Progress append flow: seed a MainItem with two SubItems (weight=1 each); append progress (completion=60) to SubItem1 → verify `SubItem1.Completion = 60` and `MainItem.Completion = 30` (avg of 60 and 0)
- Append progress (completion=80) to SubItem2 → verify `MainItem.Completion = 70` (avg of 60 and 80)
- Regression check integration: append completion=50 to SubItem1 (previous=60) → 422 `PROGRESS_REGRESSION`; `MainItem.Completion` unchanged
- ItemPool assign transaction: seed a pool item and a main item; call assign → verify `ItemPool.Status = 已分配` AND a new SubItem exists under the main item in the same DB state
- ItemPool assign rollback: if the SubItem creation is forced to fail (e.g. missing required field), verify `ItemPool.Status` remains `待分配`
- Weekly report integration: seed data for a week, call `GET /reports/weekly/export`, assert response is `text/markdown` and contains expected main item title

## Implementation Notes
- Reuse `setupTestDB()` and `setupTestRouter()` helpers from task 11.6
- Each test seeds its own data and uses a fresh DB
- For the rollback test, temporarily pass an invalid `mainItemId` (non-existent) to trigger the failure path
