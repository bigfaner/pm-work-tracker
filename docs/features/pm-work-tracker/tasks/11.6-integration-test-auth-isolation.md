# 11.6: Backend Integration Tests: Auth Flow + Team Isolation

## Description
Write integration tests using an in-memory SQLite database covering the full auth flow and team data isolation enforcement.

## Reference Files
- `design/tech-design.md` — Testing Strategy / Integration Tests
- `design/tech-design.md` — Team Isolation Enforcement section
- `design/tech-design.md` — Security / JWT Flow

## Acceptance Criteria
- Test setup: run migrations on in-memory SQLite, seed two users (userA, userB) and two teams (teamA with userA as PM, teamB with userB as PM)
- Auth flow test: `POST /auth/login` with correct credentials → 200 + token; use token on protected route → 200; use expired token → 401; no token → 401
- Team isolation test: userA authenticates, attempts `GET /teams/:teamBId/main-items` → 403 `NOT_TEAM_MEMBER`
- Team isolation test: userA can access `GET /teams/:teamAId/main-items` → 200
- SuperAdmin bypass test: superadmin user can access both teams' data
- Logout test: `POST /auth/logout` → 200 (stateless, always succeeds)
- Tests use `httptest.NewRecorder()` and `gin.Engine` directly (no real HTTP server)

## Implementation Notes
- Use a shared test helper `setupTestDB()` that returns a `*gorm.DB` with migrations applied
- Use `setupTestRouter(db)` that wires up the full router with real services (not mocks)
- Seed users with bcrypt-hashed passwords at cost 4 (faster for tests)
- Each test function gets a fresh in-memory DB to avoid state leakage
