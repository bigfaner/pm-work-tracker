# 4.3: SubItemService Implementation

## Description
Implement all `SubItemService` methods: Create, Update, ChangeStatus (with delay-count logic and auto-upgrade), Assign, List, Get.

## Reference Files
- `design/tech-design.md` — Interfaces: SubItemService
- `design/tech-design.md` — Key Constraints Summary (DelayCount >= 2 → IsKeyItem=true, Priority="P1")
- `prd/prd-spec.md` — Section 5.4 (state transition diagram, 8 status values)
- `design/tech-design.md` — Testing Strategy (ChangeStatus test scenarios)

## Acceptance Criteria
- `Create`: sets initial status `待开始`, `TeamID` from context, links to `MainItemID`
- `ChangeStatus`: validates transition against the allowed state machine; returns `ErrInvalidStatus` for illegal transitions
- `ChangeStatus` to `已延期`: increments `DelayCount`; if `DelayCount >= 2`, sets `IsKeyItem = true` and `Priority = "P1"`
- `ChangeStatus` to `已完成`: sets `ActualEndDate = now`
- `Assign`: updates `AssigneeID`; only PM can call (enforced at handler level, not service level)
- `List`: delegates to `SubItemRepo.List` with filter and pagination
- Unit tests: all valid status transitions pass, all invalid transitions return `ErrInvalidStatus`, delay-count increment, auto-upgrade at count=2

## Implementation Notes
- Valid status transitions (from → to):
  - 待开始 → 进行中, 已关闭
  - 进行中 → 阻塞中, 挂起, 待验收, 已延期, 已关闭
  - 阻塞中 → 进行中
  - 挂起 → 进行中, 已关闭
  - 已延期 → 进行中
  - 待验收 → 已完成, 进行中
  - 已完成 → (terminal)
  - 已关闭 → (terminal)
- After `ChangeStatus`, call `MainItemService.RecalcCompletion` if status changed to `已完成`
