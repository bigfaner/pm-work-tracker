# 6.7: ItemPool Handlers

## Description
Implement Gin handlers for all item pool endpoints: submit, list, get, assign, and reject.

## Reference Files
- `api-handbook.md` вҖ” ItemPool section (request/response schemas, error codes)
- `design/tech-design.md` вҖ” API Design Summary: ItemPool route table

## Acceptance Criteria
- `POST /teams/:teamId/item-pool`: any team member; returns 201 with ItemPool object, status `еҫ…еҲҶй…Қ`
- `GET /teams/:teamId/item-pool`: PM or member; paginated with status filter via `ItemPoolFilter`
- `GET /teams/:teamId/item-pool/:poolId`: PM or member; returns full ItemPool object
- `POST /teams/:teamId/item-pool/:poolId/assign`: PM only; body `{ "mainItemId": 1, "assigneeId": 3 }`; returns `{ "subItemId": 123 }`
- `POST /teams/:teamId/item-pool/:poolId/assign` returns 422 `ITEM_ALREADY_PROCESSED` if not `еҫ…еҲҶй…Қ`
- `POST /teams/:teamId/item-pool/:poolId/reject`: PM only; body `{ "reason": "..." }`; reason required
- `POST /teams/:teamId/item-pool/:poolId/reject` returns 422 `ITEM_ALREADY_PROCESSED` if not `еҫ…еҲҶй…Қ`
- Response shape matches `api-handbook.md` Data Contracts: ItemPool Object

## Implementation Notes
- `RequireTeamRole("pm")` applied to assign and reject routes
- `poolId` path param parsed as `uint`; invalid вҶ’ 400
- `submitterName` in response requires user lookup вҖ” join in service or handler
