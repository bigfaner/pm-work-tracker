# 2.2: MainItem & SubItem Models + Migrations

## Description
Define GORM model structs for `MainItem` and `SubItem`, and write the corresponding SQL migration.

## Reference Files
- `design/tech-design.md` — Data Models: MainItem, SubItem structs
- `design/tech-design.md` — Key Constraints Summary (DelayCount auto-upgrade rule, Completion weighted avg)
- `prd/prd-spec.md` — Section 5.3 (MainItem fields), Section 5.4 (SubItem fields, 8 status values)

## Acceptance Criteria
- `internal/model/main_item.go` defines `MainItem` with all fields and GORM tags from the spec
- `internal/model/sub_item.go` defines `SubItem` with all fields including `Weight float64` (default 1.0)
- `migrations/002_main_sub_items.sql` creates `main_items` and `sub_items` tables with all columns and indexes
- `MainItem.Code` has `uniqueIndex` and `size:10`
- `SubItem.Status` default is `待开始`; all 8 status values are documented in a Go comment
- Composite indexes present: `(team_id, status)`, `(team_id, priority)` on both tables

## Implementation Notes
- SubItem status enum (8 values): `待开始`, `进行中`, `阻塞中`, `挂起`, `已延期`, `待验收`, `已完成`, `已关闭`
- `MainItem.ArchivedAt *time.Time` — non-null means archived; use this instead of a separate status value
- `SubItem.DelayCount` is incremented by `SubItemService.ChangeStatus` when transitioning to `已延期`
