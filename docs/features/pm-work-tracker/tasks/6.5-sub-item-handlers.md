# 6.5: SubItem Handlers

## Description
Implement Gin handlers for all sub-item endpoints: create, list, get, update, status change, and assignee update.

## Reference Files
- `api-handbook.md` — SubItems section (request/response schemas, error codes)
- `design/tech-design.md` — API Design Summary: SubItems route table

## Acceptance Criteria
- `POST /teams/:teamId/main-items/:mainId/sub-items`: PM or any member; returns 201 with full SubItem object
- `GET /teams/:teamId/main-items/:mainId/sub-items`: team member; paginated with SubItemFilter
- `GET /teams/:teamId/sub-items/:itemId`: team member; returns full SubItem object
- `PUT /teams/:teamId/sub-items/:itemId`: PM or assignee; partial update (title, description, priority, expectedEndDate)
- `PUT /teams/:teamId/sub-items/:itemId/status`: PM or assignee; returns 422 `INVALID_STATUS` on illegal transition
- `PUT /teams/:teamId/sub-items/:itemId/assignee`: PM only; updates assigneeId
- All path params parsed as `uint`; invalid format → 400
- Response shape matches `api-handbook.md` Data Contracts: SubItem Object

## Implementation Notes
- For `PUT /:itemId` (update), only PM or the current assignee may edit — check `callerID == item.AssigneeID || callerTeamRole == "pm"` in handler
- `PUT /:itemId/status` body: `{ "status": "进行中" }`
- `PUT /:itemId/assignee` body: `{ "assigneeId": 123 }`
- `mainId` path param is validated to exist before creating sub-item
