# 1.4: Shared Error Types

## Description
Define the `AppError` struct and all predefined error variables used across the service and handler layers. Implement a Gin helper that writes the standard error envelope response.

## Reference Files
- `design/tech-design.md` — Error Handling section (AppError struct, all error vars, HTTP status mapping)
- `design/tech-design.md` — Response Envelope section

## Acceptance Criteria
- `internal/pkg/errors/errors.go` defines `AppError` struct with `Code`, `Message`, `Status` fields
- All 10 predefined error vars are present: `ErrUnauthorized`, `ErrForbidden`, `ErrTeamNotFound`, `ErrItemNotFound`, `ErrNotTeamMember`, `ErrInvalidStatus`, `ErrArchiveNotAllowed`, `ErrProgressRegression`, `ErrValidation`, `ErrInternal`
- `AppError` implements the `error` interface
- A `RespondError(c *gin.Context, err error)` helper correctly maps `*AppError` to its HTTP status and writes `{"code": "...", "message": "..."}`
- A `RespondOK(c *gin.Context, data any)` helper writes `{"code": 0, "data": ...}`
- Unit test verifies each predefined error has the correct HTTP status code

## Implementation Notes
- `RespondError` should handle both `*AppError` and generic `error` (fallback to `ErrInternal`)
- Keep error codes as string constants, not iota, to match the JSON contract in `api-handbook.md`
