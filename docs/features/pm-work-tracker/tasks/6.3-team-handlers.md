# 6.3: Team Handlers

## Description
Implement Gin handlers for all team endpoints: CRUD, member management, and PM transfer.

## Reference Files
- `api-handbook.md` — Teams section (all request/response schemas and error codes)
- `design/tech-design.md` — API Design Summary: Teams route table

## Acceptance Criteria
- `POST /teams`: requires `CanCreateTeam=true` (check from JWT claims); calls `TeamService.CreateTeam`; returns 201
- `GET /teams`: SuperAdmin gets all teams; authenticated user gets own teams
- `GET /teams/:teamId`: returns team detail with pmDisplayName, memberCount, mainItemCount
- `PUT /teams/:teamId`: requires PM role; updates name/description
- `DELETE /teams/:teamId`: requires PM or SuperAdmin; validates `confirmName` in request body
- `GET /teams/:teamId/members`: returns member list
- `POST /teams/:teamId/members`: requires PM; invites by username
- `DELETE /teams/:teamId/members/:userId`: requires PM
- `PUT /teams/:teamId/pm`: requires PM; transfers PM role to `newPmUserId`
- All error codes from `api-handbook.md` are returned correctly (ALREADY_MEMBER, CANNOT_REMOVE_SELF, etc.)

## Implementation Notes
- `CanCreateTeam` check for `POST /teams` reads from `userID` in context, then fetches user from DB (or embed in JWT claims)
- Use `RequireTeamRole("pm")` middleware on write routes within `:teamId` group
- `DELETE /teams/:teamId` body is parsed via `c.ShouldBindJSON` — confirm Gin supports body on DELETE (it does)
