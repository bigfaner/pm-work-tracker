# 3.3: Migrate Hardcoded Colors to Theme Tokens

## Description

Replace all 58 hardcoded Tailwind color classes with theme tokens. Also update the `--color-success` token family from green to blue as specified in tech design.

**Color mapping**:
| Hardcoded | Token |
|-----------|-------|
| `emerald-500/600/700` | `success` / `success-text` / `success-bg` |
| `red-400/500/600/800` | `error` / `error-text` / `error-bg` |
| `amber-300/500/600/800` | `warning` / `warning-text` / `warning-bg` |
| `slate-*` | `secondary` / `tertiary` |

**Token value change** (in `src/index.css`):
| Token | Current (green) | Target (blue) |
|-------|----------------|---------------|
| `--color-success` | `#059669` | `#3b82f6` |
| `--color-success-bg` | `#ecfdf5` | `#eff6ff` |
| `--color-success-text` | `#047857` | `#1d4ed8` |

**Migration order**: ui/ components (badge, button, toast) → shared components → pages

## Reference Files

- Source: `docs/features/code-conventions/design/tech-design.md` (Section 4: Color Token Mapping)
- Theme tokens: `frontend/src/index.css`
- UI components: `frontend/src/components/ui/`
- Search: grep for `emerald-`, `red-`, `amber-`, `slate-` in `.tsx` files

## Acceptance Criteria

- [ ] 0 hardcoded color classes (emerald-*, red-*, amber-*, slate-*) in `.tsx` files
- [ ] Theme tokens updated: success family changed to blue
- [ ] ESLint color restriction rule passes with 0 violations
- [ ] All frontend tests pass
- [ ] No visual regression (colors map correctly to theme)

## Implementation Notes

- Start with `src/components/ui/` (badge, button, toast)
- Then shared components, then pages
- Run tests after each file to catch regressions
- Be careful with `slate-*` — some may be intentional (not all slate is wrong)
- Check `--color-success` CSS variable in index.css for the value change
