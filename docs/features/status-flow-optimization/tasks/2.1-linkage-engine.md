# 2.1: Linkage Engine — EvaluateLinkage & SubItem Integration

## Description

Implement the main-sub item linkage: `EvaluateLinkage` method with 5-level priority rules, per-MainItem mutex, `LinkageResult` struct, and integration into SubItem's ChangeStatus/Create/Delete. Add linkageWarning to API responses.

## Reference Files

- Design: "LinkageResult Struct", "New Method: EvaluateLinkage", "Concurrency: Per-MainItem Mutex", "Modified Service: SubItemService", "Linkage Failure Toast", "Testing Strategy" sections
- Modified: `backend/internal/service/main_item_service.go`
- Modified: `backend/internal/service/sub_item_service.go`
- Modified: `backend/internal/handler/sub_item_handler.go`

## Acceptance Criteria

- [ ] `LinkageResult` struct defined with Triggered, Success, TargetStatus, Remark fields and `Warning()` method
- [ ] Per-MainItem mutex implemented: `getLinkageMutex(mainItemID)` function with `sync.Mutex` map
- [ ] `EvaluateLinkage` acquires mutex, fetches SubItems, evaluates priority rules, updates MainItem
- [ ] Priority 1: all completed/closed + at least one completed → reviewing (AC-7)
- [ ] Priority 2: all closed → closed
- [ ] Priority 3: all pausing (or pausing + closed) → pausing
- [ ] Priority 4: any blocking (not all terminal) → blocking (only from pending/progressing)
- [ ] Priority 5: any progressing → progressing (only from pending)
- [ ] All 5 priority levels have dedicated test cases (AC-8)
- [ ] reviewing + new pending subitem → MainItem reverts to progressing (AC-9)
- [ ] Delete subitem triggers re-evaluation (AC-10)
- [ ] No subitems → no linkage triggered (AC-11)
- [ ] Linkage failure: MainItem unchanged, StatusHistory records intent + remark (AC-12)
- [ ] Terminal side effects applied when linkage transitions to terminal
- [ ] StatusHistory records: is_auto=true for linkage, is_auto=false for manual (AC-15)
- [ ] RecalcCompletion runs before EvaluateLinkage on SubItem completed (AC-13)
- [ ] SubItem.ChangeStatus populates LinkageResult in SubItemChangeResult
- [ ] SubItem handler includes `linkageWarning` in API response when Warning() is non-empty (AC-12 toast)
- [ ] Service tests for all linkage scenarios pass

## Implementation Notes

- Fixed execution order in SubItem.ChangeStatus: (1) RecalcCompletion, (2) EvaluateLinkage
- SubItem.Create: after creating, call `EvaluateLinkage(ctx, item.MainItemID, callerID)`
- SubItem.Delete: after deleting, call `EvaluateLinkage(ctx, item.MainItemID, callerID)`
- EvaluateLinkage returns `(*LinkageResult, error)` — nil result when no linkage triggered
- Handler: check `result.LinkageResult.Warning()`, include as `linkageWarning` field in response data
- Mutex map: no cleanup needed (bounded by MainItem count)
