# 1.1: Status Enum Package

## Description

Create `pkg/status` package as the single source of truth for status definitions and transition rules. This is a pure data package with no database or service dependencies.

## Reference Files

- Design: `docs/features/status-flow-optimization/design.md` — "New Package: pkg/status"
- New: `backend/internal/pkg/status/status.go`
- New: `backend/internal/pkg/status/transition.go`
- New: `backend/internal/pkg/status/status_test.go`
- New: `backend/internal/pkg/status/transition_test.go`

## Acceptance Criteria

- [ ] `MainItemStatuses` map contains 7 entries: pending, progressing, blocking, pausing, reviewing, completed, closed
- [ ] `SubItemStatuses` map contains 6 entries: pending, progressing, blocking, pausing, completed, closed
- [ ] Each StatusDef has correct Code, Name (Chinese), and Terminal fields
- [ ] `MainItemTransitions` defines exactly 10 legal transition paths (PRD AC-2)
- [ ] `SubItemTransitions` defines exactly 9 legal transition paths (PRD AC-3)
- [ ] `IsValidTransition` returns true for all legal transitions, false for illegal/self/unknown
- [ ] `GetAvailableTransitions` returns correct target list per status; empty for terminal/unknown
- [ ] `IsMainTerminal` / `IsSubTerminal` return true only for completed/closed
- [ ] Table-driven tests cover all transition combinations with ≥90% coverage

## Implementation Notes

- Package name: `status` (lowercase, single word per naming rules)
- No imports from model/service/handler packages
- `IsValidTransition` checks `transitions[from]` contains `to`; returns false if `from == to`
- `GetAvailableTransitions` returns a copy of the slice (not the original) to prevent mutation
