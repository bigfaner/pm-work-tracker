# Cache-Control 评估：`max-age=31536000, immutable` 在 CDN 链路下的影响

**请求链路：** 浏览器 → CDN Edge → (CDN Mid-tier) → Origin（Go 二进制）

## 各层行为

| 层 | 行为 |
|----|------|
| 浏览器 | 1 年内不发任何请求（含 F5 刷新），`immutable` 阻止条件请求 |
| CDN Edge | 缓存响应 1 年，同区域用户命中边缘缓存，不回源 |
| CDN Mid-tier | 同上，进一步减少回源 |

## 安全性前提

文件名含 content hash（如 `app.abc123.js`），内容变更 → 文件名变更 → 产生新 URL。旧 URL 对应的内容永远不变，激进缓存安全。

`index.html` 走 `no-cache`，每次拉取最新 → 引用新 hash 文件名 → 用户自动获取新版本。

## 风险场景

| 场景 | 影响 | 是否可接受 |
|------|------|-----------|
| **回滚部署** | CDN 缓存了 v2 资源，但回滚后 origin 只提供 v1。用户拿到 v1 的 `index.html`（`no-cache` 保证），引用 v1 的 hash 文件名，v1 资源要么在 CDN 缓存中，要么从 origin 获取 | 可接受，`no-cache` 入口页保证正确路由 |
| **紧急修复 JS bug** | 无法就地修复，必须发布新 hash 文件名。旧缓存资源用户不受影响（已加载的页面继续运行） | 可接受，hash 策略的设计意图 |
| **CDN 节点故障/清空** | 所有请求回源，Go 二进制直接承受全部静态资源流量 | 低风险，单开发者场景流量小 |
| **hash 碰撞或不确定** | 不同内容产出相同文件名 → 用户拿到旧内容 | 极低概率，Vite 使用内容 hash |

## 结论

对当前项目（单开发者、低频部署、单服务器），`max-age=31536000, immutable` 是正确选择。CDN 加入后收益放大（减少回源、降低延迟），风险可控（`index.html` 的 `no-cache` 是安全锚点）。

**注意：** 未来接入 CDN 时，确认 CDN 是否尊重 origin 的 `Cache-Control`。部分 CDN 有自己的缓存策略覆盖机制，需在 CDN 配置中显式"遵循源站"。这属于 CDN 配置范畴，不影响当前响应头设计。
