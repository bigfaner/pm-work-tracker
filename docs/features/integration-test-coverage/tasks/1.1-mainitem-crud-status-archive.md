# 1.1: MainItem CRUD + Status + Archive Integration Tests

## Description

Write integration tests for 10 MainItem endpoints: create, list, detail, update, status change, available transitions, and archive. Tests cover happy path, validation errors (422), permission denial (403), not-found (404), and cascade effects (terminal status propagation to SubItems).

Uses existing inline helpers from `auth_isolation_test.go` (setupTestRouter, loginAs, signTokenWithClaims) and `rbac_test.go` (makeRequest) since helpers.go does not exist yet.

## User Stories

### Story 1: Item Lifecycle (partial — MainItem endpoints)

- **AC**: Given a team and PM user, When PM creates MainItem with valid fields, Then returns 201 and record is queryable in DB
- **AC**: Given an in-progress MainItem, When PM requests valid status transition, Then status changes and SubItems auto-marked terminal
- **AC**: Given an in-progress MainItem, When PM requests invalid transition (e.g., new→completed), Then returns 422 and status unchanged
- **AC**: Given a completed MainItem, When PM archives, Then returns 200; when item is in-progress, archive returns 422
- **AC**: Given a member role user, When attempting create/update/change status MainItem, Then returns 403

## Implementation Notes

- Use `setupTestRouter(t)` to get router + seedData (userA is PM of teamA)
- Use `loginAs(t, r, "userA", "passwordA")` for PM token
- Use `signTokenWithClaims(t, claims)` for member token (memberAID from seedData)
- Use `makeRequest(t, r, method, path, body, token)` for HTTP calls
- Test naming: `TestItemLifecycle_<Action>_<ExpectedOutcome>`
- Each test gets its own DB via `setupTestRouter` (per-test isolation)
- Assert on HTTP status codes and response body structure (JSON field existence), not on field order
- For status transitions, reference `status/transition.go`: pending→progressing/closed, progressing→blocking/pausing/reviewing/closed, etc.
