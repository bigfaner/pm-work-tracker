# 1.2: SubItem CRUD + Progress + Assignee Integration Tests

## Description

Write integration tests for 7 SubItem endpoints: create sub-item, list, detail, update, status change, available transitions, assignee management, progress append, progress list, and progress completion correction. Tests cover happy path, validation errors, permission denial, not-found, and cascade effects (completion rollup to MainItem, auto status transition on 100%).

## User Stories

### Story 1: Item Lifecycle (SubItem + Progress endpoints)

- **AC**: Given an existing MainItem, When PM creates SubItem with weight, Then SubItem links to MainItem and MainItem completion stays correct
- **AC**: Given a 0% SubItem, When PM appends Progress (completion=60), Then SubItem completion updates to 60 and MainItem completion recalculates by weight
- **AC**: Given SubItem at 100% completion, When status auto-transitions to completed, Then MainItem completion reflects this

## Implementation Notes

- Reuse `seedProgressData(t, db, teamID, userID)` from `progress_completion_test.go` for tests needing pre-seeded MainItem + SubItems
- For tests needing a single MainItem, create via HTTP (POST /teams/:id/main-items) using PM token
- SubItem status machine differs from MainItem: can go progressing→completed directly (MainItem must go reviewing→completed)
- Progress regression: appending completion < last value returns 422 with error code `PROGRESS_REGRESSION`
- Completion rollup: when SubItem progress changes, MainItem completion = weighted average of all SubItems
- Test naming: `TestItemLifecycle_<SubAction>_<ExpectedOutcome>`
