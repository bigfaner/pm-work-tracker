# 3.3: Admin User Integration Tests (F4)

## Description

Write integration tests for 6 Admin User Management endpoints: list users, create user, user detail, update user, toggle user status, list teams. Tests cover happy path, validation errors, SuperAdmin-only access (403), duplicate username (409), self-disable protection (422), and not-found (404).

## Reference Files

- `docs/features/integration-test-coverage/design/tech-design.md` — Test Request Pattern, Error Assertion Pattern
- `docs/features/integration-test-coverage/prd/prd-spec.md` — F4 endpoint test matrix
- `backend/tests/integration/helpers.go` — shared helpers
- `backend/internal/handler/admin_handler.go` — handler implementation
- `backend/internal/service/admin_service.go` — business logic

## Acceptance Criteria

- [ ] `admin_user_test.go` created in `backend/tests/integration/`
- [ ] `GET /admin/users` — paginated list with search returns 200, non-SuperAdmin returns 403
- [ ] `POST /admin/users` — create returns 201, duplicate username returns 409, non-SuperAdmin returns 403
- [ ] `GET /admin/users/:userId` — detail returns 200, not-found returns 404
- [ ] `PUT /admin/users/:userId` — update returns 200, empty displayName / displayName > 64 / email > 100 / invalid teamKey returns 422, not-found returns 404
- [ ] `PUT /admin/users/:userId/status` — toggle enable/disable returns 200, disable self returns 422, not-found returns 404
- [ ] `GET /admin/teams` — team list with member count returns 200
- [ ] All tests pass: `go test ./tests/integration/ -run TestAdminUser`

## User Stories

### Story 4: Admin User Management Flow

- **AC**: Given a SuperAdmin, When creating user (valid username + password), Then returns 201
- **AC**: Given a SuperAdmin, When creating user with existing username, Then returns 409
- **AC**: Given a SuperAdmin, When disabling a user, Then user status updates to disabled
- **AC**: Given a SuperAdmin, When attempting to disable self, Then returns 422
- **AC**: Given a regular user, When accessing /admin/users, Then returns 403

## Implementation Notes

- Use `loginAs(t, r, "superAdmin", "password")` or `signTokenWithClaims` with superAdminRoleID for SuperAdmin token
- For non-admin 403 tests: use regular user token (userA or memberA)
- Self-disable test: SuperAdmin disables their own user ID → 422
- Duplicate username test: create a user, then create another with same username → 409
- For teamKey validation in PUT: pass a teamKey that doesn't correspond to any team → 422
- Test naming: `TestAdminUser_<Action>_<ExpectedOutcome>`
