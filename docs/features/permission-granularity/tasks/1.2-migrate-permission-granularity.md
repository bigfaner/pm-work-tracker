# 1.2: 新增 MigratePermissionGranularity 迁移函数

## Description

在 `backend/internal/migration/rbac.go` 中新增 `MigratePermissionGranularity` 函数，将存量 `pmw_role_permissions` 中的旧权限码转换为新权限码。函数必须幂等，通过 `schema_migrations` 表追踪版本。

迁移逻辑（事务内执行）：
1. 快照备份（记录受影响的 role_id 列表）
2. `user:manage_role` → `role:create` + `role:update` + `role:delete`（INSERT IGNORE，DELETE 旧码）
3. 旧 `user:read` → INSERT `user:list`（旧 `user:read` 行保留，语义随路由绑定更新）
4. 标记 `schema_migrations` version = `permission_granularity_001`

同时更新 `seedPresetRoles` 中 pm 角色的权限码列表（新增 `user:list`、`user:read`、`user:assign_role`、`role:*`，移除 `user:manage_role`）。

## Reference Files

- `backend/internal/migration/rbac.go` — 迁移基础设施（参考 `MigrateToRBAC` 实现模式）
- `docs/features/permission-granularity/design/tech-design.md` §Interfaces 3、4 — 迁移函数规格

## Acceptance Criteria

- [ ] `MigratePermissionGranularity(db)` 函数存在，签名为 `func MigratePermissionGranularity(db *gorm.DB) error`
- [ ] 执行后，持有 `user:manage_role` 的角色改为持有 `role:create`、`role:update`、`role:delete`，旧码不再存在
- [ ] 执行后，持有旧 `user:read` 的角色同时持有 `user:list`（旧 `user:read` 行保留）
- [ ] 重复执行无副作用（幂等）
- [ ] 事务失败时自动回滚，无部分迁移残留
- [ ] `seedPresetRoles` 中 pm 角色包含新权限码列表，不含 `user:manage_role`
- [ ] `go test ./backend/internal/migration/ -run TestMigratePermissionGranularity` 通过（新增 3 个测试用例）

## User Stories

Story 5 AC:
- Given 数据库中存在持有 `user:manage_role` 的自定义角色，When 迁移执行完成，Then 该角色持有 `role:create+update+delete`，原码不再存在
- Given 数据库中存在持有旧 `user:read` 的角色，When 迁移执行完成，Then 该角色同时持有 `user:list`
- Given 迁移过程中发生错误，When 事务回滚，Then 数据库恢复到迁移前状态

## Implementation Notes

- 参考 `MigrateToRBAC` 的 `ensureSchemaMigrationsTable` + version check 模式
- 调用方（`main.go` 或 `cmd/server`）在 `MigrateToRBAC` 之后、`SyncPresetRoles` 之前调用此函数；若返回非 nil error，必须终止启动并记录日志
- 新增测试文件 `backend/internal/migration/permission_granularity_test.go`
