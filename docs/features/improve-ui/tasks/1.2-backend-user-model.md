# 1.2: Backend User Model Extension

## Description

Extend the User model with `Email` and `Status` fields, create the database migration, and update related DTO/auth logic.

**Scope:**
1. Add `Email string` and `Status string` fields to `backend/internal/model/user.go`
2. Create migration file `backend/migrations/004_user_email_status.sql`
3. Update `backend/internal/dto/auth.go` — add `Email`, `Status`, `Teams` fields to UserDTO
4. Update auth service to check user status during login (disabled → `USER_DISABLED` error)
5. Register new error codes in `backend/internal/pkg/errors/` (USER_EXISTS, USER_NOT_FOUND, USER_DISABLED, INVALID_STATUS, CANNOT_DISABLE_SELF, TEAM_NOT_FOUND, FUTURE_WEEK_NOT_ALLOWED)
6. Update existing user queries to include new fields

## Reference Files

- `backend/internal/model/user.go` — User model
- `backend/internal/dto/auth.go` — UserDTO definition
- `backend/internal/service/auth_service.go` — login logic
- `backend/internal/pkg/errors/errors.go` — error code registry
- `backend/migrations/` — existing migration pattern
- `docs/features/improve-ui/design/tech-design.md` — Data Models section
- `docs/features/improve-ui/design/api-handbook.md` — error codes

## Acceptance Criteria

- [ ] User model has `Email` (size:100) and `Status` (size:10, default:"enabled") fields
- [ ] Migration file `004_user_email_status.sql` exists with correct ALTER TABLE statements
- [ ] Login endpoint returns `USER_DISABLED` (403) when a disabled user tries to log in
- [ ] All new error codes registered in errors package with correct HTTP status mappings
- [ ] Existing tests still pass

## Implementation Notes

- The `Status` field default value is "enabled". Existing users should not be affected.
- Follow the existing migration pattern (see `backend/migrations/001_init_test.go` for how migrations are tested).
- Error codes should follow the existing `AppError` pattern in `errors/errors.go`.
