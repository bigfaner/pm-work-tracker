# 3.1: Admin User CRUD Endpoints

## Description

Implement 4 new admin endpoints for user management and enhance the existing user list endpoint.

**New endpoints:**
1. `POST /admin/users` — create user (auto-generate 12-char random password, return `initialPassword` once)
2. `GET /admin/users/:userId` — get single user details (with teams)
3. `PUT /admin/users/:userId` — update user (displayName, email, canCreateTeam, teamId)
4. `PUT /admin/users/:userId/status` — toggle enable/disable status

**Enhanced endpoint:**
5. `GET /admin/users` — add search, canCreateTeam filter, pagination params; add email, status, teams to response

**Backend changes:**
- `AdminService` — add CreateUser, GetUser, UpdateUser, ToggleUserStatus methods
- `AdminHandler` — add handler methods for new endpoints
- `router.go` — register new routes under `/api/v1/admin/`
- `UserRepo` — add new query methods (FindByUsername, FindByID, etc.)
- Password generation: 12 chars (uppercase + lowercase + digits), bcrypt hash for storage

## Acceptance Criteria

- [ ] POST /admin/users creates user with auto-generated password, returns 201 with `initialPassword` field
- [ ] GET /admin/users/:userId returns user details with teams array
- [ ] PUT /admin/users/:userId updates user fields, returns updated user
- [ ] PUT /admin/users/:userId/status toggles between enabled/disabled
- [ ] Cannot disable self (returns CANNOT_DISABLE_SELF 422)
- [ ] Duplicate username returns USER_EXISTS (422)
- [ ] Invalid userId returns USER_NOT_FOUND (404)
- [ ] GET /admin/users supports search, canCreateTeam filter, pagination with new fields (email, status, teams)
- [ ] All endpoints require SuperAdmin auth
- [ ] Existing admin tests still pass

## Implementation Notes

- Password generation: use `crypto/rand` for secure random bytes, charset [A-Za-z0-9], length 12.
- The `initialPassword` is returned in the response only on creation. Never stored in plaintext — only bcrypt hash.
- User teams: query through TeamMember join table to get user's team memberships.
- Follow existing handler/service/repo patterns exactly (see admin_handler.go, admin_service.go).
