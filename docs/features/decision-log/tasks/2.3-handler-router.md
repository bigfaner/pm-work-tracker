# 2.3: Implement DecisionLog Handler and Register Routes

## Description

Implement the DecisionLogHandler HTTP methods (Create, Update, Publish, List) and register decision log routes nested under the main-items resource group.

## Reference Files

- `design/tech-design.md` — Handler interface, Security Considerations (Route Registration), Cross-Layer Data Map
- `design/api-handbook.md` — Endpoint specs, request/response formats, error codes
- `backend/internal/handler/main_item_handler.go` — Reference for handler patterns
- `backend/internal/router/router.go` — Route registration location

## Acceptance Criteria

- [ ] `Create` handler extracts teamID, mainItemID, userID from context, binds `DecisionLogCreateReq`, calls service, returns 201 with VO
- [ ] `Update` handler extracts logId from URL, resolves BizKey to internal ID, binds `DecisionLogUpdateReq`, calls service, returns 200 with VO
- [ ] `Publish` handler extracts logId from URL, calls service, returns 200 with VO
- [ ] `List` handler extracts pagination params, applies `dto.ApplyPaginationDefaults`, calls service, returns 200 with PageResult
- [ ] VO conversion includes `CreatorName` resolution via batch user lookup (same pattern as progress records)
- [ ] All errors caught and passed to `apperrors.RespondError(c, err)`
- [ ] Routes registered in `router.go` inside `teamsGroup` block
- [ ] Write routes gated by `deps.perm("main_item:update")`
- [ ] Read route (List) requires team membership only
- [ ] Route paths match api-handbook spec: `/main-items/:itemId/decision-logs[/:logId[/publish]]`
- [ ] Handler tests pass: `go test ./internal/handler/ -run TestDecisionLog`

## User Stories

- **Story 1 (记录决策)**: POST endpoint creates decision log and returns VO
- **Story 2 AC (Edit draft)**: PUT endpoint updates draft, service enforces ownership
- **Story 2 AC (Publish from draft)**: PATCH endpoint publishes draft
- **Story 3 (查阅历史决策)**: GET endpoint returns paginated decision list
- **Story 4 AC (Published edit → 403)**: Service returns 403, handler passes through
- **Story 4 AC (No permission → no add)**: Route middleware enforces `main_item:update` for write operations
- **PRD 5.2 (Permission = main_item:update)**: Write routes use `deps.perm("main_item:update")`

## Implementation Notes

- Follow route registration pattern from Security Considerations section of tech-design
- Use `ResolveBizKey` for URL param → internal ID conversion
- Use `middleware.GetTeamID()` / `middleware.GetUserID()` for auth context extraction
- VO batch user lookup: collect all `CreatedBy` IDs from list results, single batch query via `userRepo`, map to names
- Wire handler in dependency container (e.g., `deps.DecisionLog`)
- `NewDecisionLogHandler` takes `DecisionLogService`, `UserRepo`, `MainItemRepo` as params
