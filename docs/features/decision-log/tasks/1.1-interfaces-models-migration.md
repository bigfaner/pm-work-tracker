# 1.1: Define Interfaces, Data Model, Error Types, and DB Migration

## Description

Define all foundational artifacts for the DecisionLog feature: repository/service/handler interfaces, GORM model, VO, DTOs, error types, and database migration for both SQLite and MySQL.

## Reference Files

- `design/tech-design.md` — Interfaces, Data Models, Cross-Layer Data Map, Error Handling sections
- `design/api-handbook.md` — API contracts and request/response shapes
- `backend/internal/model/progress_record.go` — Reference for BizKey pattern (no BaseModel)
- `backend/internal/model/main_item.go` — Reference for existing model patterns
- `backend/migrations/SQLite-schema.sql` — Schema file to update
- `backend/migrations/MySql-schema.sql` — Schema file to update

## Acceptance Criteria

- [ ] `DecisionLogRepo` interface defined in `backend/internal/repository/decision_log_repo.go` with Create, FindByID, FindByBizKey, ListByItem, Update methods
- [ ] `DecisionLogService` interface defined in `backend/internal/service/decision_log_service.go` with Create, Update, Publish, List methods
- [ ] `DecisionLogHandler` struct and constructor defined in `backend/internal/handler/decision_log_handler.go`
- [ ] `DecisionLog` model defined in `backend/internal/model/decision_log.go` with TableName "pmw_decision_logs", all fields matching tech-design
- [ ] `DecisionLogVO` defined with Tags as `[]string`, CreatorName field, all BizKeys formatted as strings
- [ ] `DecisionLogCreateReq` and `DecisionLogUpdateReq` DTOs defined with binding validations
- [ ] `ErrDecisionLogNotFound` added to apperrors
- [ ] `pmw_decision_logs` table added to both SQLite and MySQL schema files
- [ ] Project compiles: `cd backend && go build ./...`
- [ ] Existing tests pass: `cd backend && go test ./...`

## User Stories

- **Story 1 (记录决策)**: DecisionLog model stores category, tags, content, and status. CreateReq accepts all required fields.
- **Story 2 (草稿管理)**: logStatus field supports "draft" and "published" values. Model default is "draft".
- **Story 4 (权限隔离)**: Model has CreatedBy field for ownership tracking. ListByItem interface supports userID filtering for drafts.

## Implementation Notes

- Follow ProgressRecord pattern: own BizKey, no BaseModel embedding
- `Tags` field: store as JSON string in TEXT column, parsed to `[]string` in VO layer
- `LogStatus` column name avoids MySQL reserved word `status` — follows `item_status`/`pool_status` convention
- `MainItemKey` named explicitly (not `ItemKey`) to avoid ambiguity with StatusHistory
- `TeamKey` added for team-scoped query consistency
- Both SQLite and MySQL schema files must be updated in sync
- NewDecisionLogVO() must handle JSON string → []string conversion for Tags
- Add `FormatID()` calls for all int64 BizKey fields in VO
