# 2.1: Implement DecisionLog Repository

## Description

Implement the GORM-based DecisionLogRepo with CRUD operations and the ListByItem query that applies draft visibility filtering (published decisions visible to all + drafts visible to creator only).

## Reference Files

- `design/tech-design.md` — DecisionLogRepo interface, Data Model, Cross-Layer Data Map
- `backend/internal/repository/gorm/main_item_repo.go` — Reference for existing repo patterns
- `backend/internal/repository/decision_log_repo.go` — Interface definition (from task 1.1)

## Acceptance Criteria

- [ ] `decisionLogRepo` struct implements `DecisionLogRepo` interface in `backend/internal/repository/gorm/decision_log_repo.go`
- [ ] `Create` persists a new DecisionLog record
- [ ] `FindByID` returns record by uint ID, returns `gorm.ErrRecordNotFound` if missing
- [ ] `FindByBizKey` returns record by int64 BizKey, returns `gorm.ErrRecordNotFound` if missing
- [ ] `ListByItem` applies draft visibility filter: `(log_status = 'published' OR created_by = ?)` with `userID` parameter
- [ ] `ListByItem` returns results ordered by `create_time DESC`, with pagination via `offset`/`limit`
- [ ] `ListByItem` returns total count of matching records
- [ ] `Update` saves modified fields to existing record
- [ ] Unit tests pass: `go test ./internal/repository/gorm/ -run TestDecisionLog`

## User Stories

- **Story 1 AC (Published visible to all)**: ListByItem returns all published decisions without CreatedBy filter
- **Story 2 AC (Draft visible to creator only)**: ListByItem filters drafts by `created_by = userID`
- **Story 3 (查阅历史决策)**: ListByItem orders by `create_time DESC` for reverse chronological display

## Implementation Notes

- Follow existing repository patterns (e.g., `mainItemRepo`, `progressRecordRepo`)
- Use `repo.FindByID` generic helper for FindByID
- ListByItem query: `WHERE main_item_key = ? AND (log_status = 'published' OR created_by = ?) ORDER BY create_time DESC LIMIT ? OFFSET ?`
- Count query shares the same WHERE clause
- Tags stored as JSON string — no special handling at repo layer (raw string pass-through)
