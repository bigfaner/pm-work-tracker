# User Stories: 决策日志

## Story 1: 记录决策

**As a** PM / 技术负责人
**I want to** 在主事项详情页记录决策（选择分类、添加标签、填写内容）
**So that** 团队决策有系统化的记录，新成员可自助查阅历史决策背景

**Acceptance Criteria:**
- Given 我在主事项详情页且有编辑权限
- When 我点击「添加决策」按钮并填写分类和内容后点击「发布」
- Then 决策以已发布状态出现在时间线中，所有团队成员可见，且不可修改

---

## Story 2: 草稿管理

**As a** PM / 技术负责人
**I want to** 将决策保存为草稿并反复编辑后再发布
**So that** 我可以逐步完善决策内容，确保发布前信息准确完整

**Acceptance Criteria:**
- Given 我在决策表单中填写了分类和内容
- When 我点击「保存草稿」
- Then 决策以草稿状态保存，时间线中显示「草稿」标记，仅我可见

- Given 我有一条草稿决策
- When 我点击编辑并修改内容后点击「保存草稿」
- Then 草稿内容更新，仍为草稿状态

- Given 我有一条草稿决策
- When 我点击编辑后点击「发布」
- Then 决策变为已发布状态，所有团队成员可见，不可再编辑

---

## Story 3: 查阅历史决策

**As a** 新成员 / 其他团队成员
**I want to** 在主事项详情页按时间线查阅已发布的历史决策（含分类和标签）
**So that** 我可以自助了解事项推进过程中的关键决策背景，减少对老成员的依赖

**Acceptance Criteria:**
- Given 我在主事项详情页
- When 我查看决策记录区域
- Then 我可以看到按时间倒序排列的所有已发布决策，每条显示分类标签、自由标签、内容摘要（前 80 字符）、录入人、时间

- Given 决策内容超过 80 字符
- When 我点击该条目
- Then 展开显示完整内容

---

## Story 4: 验证权限隔离生效

**As a** 团队管理员
**I want to** 验证非授权用户无法添加决策，且草稿仅对录入人可见
**So that** 我可以确认决策记录的访问和修改受到正确控制，保护审计完整性

**Acceptance Criteria:**
- Given 用户 A 创建了一条草稿决策
- When 用户 B 查询同一主事项的决策列表
- Then 返回结果中不含用户 A 的草稿

- Given 已发布决策
- When 任何用户尝试通过 API 修改该决策
- Then 返回 403 错误

- Given 用户没有 main_item:update 权限
- When 用户查看主事项详情页
- Then 用户看不到「添加决策」按钮，但可查看已发布决策
